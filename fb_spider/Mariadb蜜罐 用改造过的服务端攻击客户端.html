Mariadb蜜罐：用改造过的服务端攻击客户端

c0debreak 2017-03-15 共206911人围观 ，发现 21 个不明物体 工具网络安全

*本文原创作者：c0debreak，本文属FreeBuf原创奖励计划，未经许可禁止转载

上周发现了一个神奇的 Mariadb 服务端插件，可以用来做蜜罐，这里分享给大家。说是一个蜜罐，但在渗透中，也可以用来搞定某些服务器，你懂的。

简介

简单讲，MariaDB 存在一个未公开的协议，在客户端进行查询前，重写客户端要执行的查询语句，并重新发起查询。那么这个有什么危害呢？

如果我们将客户端的查询语句，替换为某些恶意的语句，e.g.

LOAD DATA LOCAL INFILE '/etc/passwd' INTO TABLE test.loot

就可以实现一些对客户端的攻击。如果是Windows客户端，这个危害还可以进一步扩大，你懂的。

其实这个场景还是很多的，很多 MySQL 监控程序，都会连接数据库，执行一些语句，e.g.

SELECT @@server_id

如果被替换成读取敏感文件的语句，Well~

配置服务

我们用 Ubuntu 16.04 进行演示，安装好 mariadb 和 maxscale

MaxScale For Ubuntu 在这里下载

我们先简单配置下 Mariadb，创建一个库和用户，

确认 mariadb 可用之后，我们再配置下 maxscale 插件，完整配置在文末可以下载。

我们添加一个新的 Filter, 它负责把 `select @@server_id` 替换为 `LOAD DATA` 语句，实现客户端文件盗取。

实际效果演示

下面我们打开另外一台虚拟机，一台 CentOS 5 的服务器

我们先确认下 /etc/passwd 的内容，和 MariaDB 所在的服务器内容不同，

客户端的文件，最后一行是 xfs 用户，

而恶意的MariaDB服务器上，最后一行是 work 账号，而且没有 xfs 用户（grep 验证了下）

下面我们来连接到恶意的服务器，执行一下 select @@server_id 命令（注意 MaxScale 监听的是 4008 端口）

执行成功，回到恶意的MairaDB服务器，发现 test.loot 已经写入了数据，而且是客户端的文件哦

好了，我们成功的读取到了 /etc/passwd，那么这个真的只影响 MySQL 客户端？不是的，不管你用 PHP、Python 还是 Ruby，都会受到影响，所以这个危害还是很大的。

当然，一个聪明的攻击者，应当禁用Mariadb的认证机制，让任何客户端都能够连接。

        @ c0debreak  抱歉，我可能没表述明白，详细说明一下我的疑问，可能这方面我没有注意到，请多指教。

        1.一般的sql查询是放在服务端的frontend，而数据库是backend，应该是同个服务器，或者说是同属于服务端的功能不可能放到用户端（考虑到不可控性和非保密性）。文中的客户端写到的是直接与DB交互，我的理解应该是指代frontend而不是真正的用户端。所以产生了疑点，我有些怀疑是否能真的在Client产生执行。意思就是，理论上生产环境的查询全应该在服务器才对，所以不管什么脚本类型，应该在服务端完成SQL命令，只以API或者其他方式交互到用户手上。那么哪怕是服务前端可以造成执行打到前端服务器的数据，在插件参数不可控情况下，或者插件是否安装都不知道的情况下，理论用户是干涉不到SQL的执行流程，也就更无法替换了。

        2.光是客户端就造成执行命令，还是从本地读取文件，流化直接传入服务器backend。我理解为本地脚本执行查询，直接对暴露的backend进行交互（如文中最后一条描述）。那么在此情况下，必须在客户端就完成命令替换，理论上查询是单向操作的，就是说Clinet->Server（我对这个数据库和这个插件不太了解，抱歉），如果插件是安装在Server的那执行的时候也是Server端的文件被读取，那么如何做到读取用户本地的数据呢？

        3.用户端的数据读取，必须在本地完成，这个流程必须先于上传数据，那必须让操作脚本直接在客户端执行完毕读取文件的命令，哪怕是插件导致的命令逆流，那应该算在客户端的二次执行吧，那就假设这个plugin是客户端版的，就得让客户下载插件（或者封装在client里）然后执行，那么……我都能让客户直接执行我的脚本了……？我直接用脚本操作貌似也不需要用到插件，不是嘛……除非用户端的这个插件能再开一个后台任务，暴露端口，监听服务端返回的二次命令并在本地执行再把流内容丢回你后端。不然我真没想通如何在Server能读到Client本地的文件数据。
