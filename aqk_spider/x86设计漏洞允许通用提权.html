x86设计漏洞允许通用提权

2015-08-10 19:11:38 阅读：7816次 点赞(0) 收藏 来源： 360安全播报

x86设计漏洞允许通用提权,可能影响的所有CPU版本

内存水坑攻击技术

摘要——对于x86体系来说,以ring 0模式运行的代码不仅能够访问更多的特权内存区域,还能在AV面前做到隐形,此外,还可无拘无束地访问各种硬件,并能直抢占或修改操作系统。由于这种体系结构已经在那些“负”环之上叠放了许多的防御层,所以在x86经过四十年的发展之后,人们已经渐渐遗忘了这些进入超级特权模式的后门。这种体系结构,就像一座古老的建筑,经过长年累月的开发和修补之后,俨然已经变成了一座拜占庭式的迷宫,所以存在遗留了20年的设计漏洞也就不足为奇了。因此在x86体系中,那些被人遗忘的功能特性,婉如一片广阔的处女地,静静地等待着我们去开疆拓土。下面,我们将向大家介绍如何让恶意代码从ring 0进入处理器中最深、最暗的领域。这种攻击是通过体系结构的0-day漏洞来干掉它自己的,并且,存在这种漏洞的代码广泛存在于现代系统之中。

一、导言

过去,x86体系结构将特权分为不同的级别,每一个级别对应一个环(ring),ring 3级别拥有最低的权限,ring 0级别拥有最高的权限。随着这个体系结构不断发展,人们发现越来越需要引入更深的级别,同时还需要附加相应特权隔离机制,以便限制ring 0级别的代码获得更高的运行权限,这时,所谓的负环就应运而生了。Ring -1,更常见的称谓是系统管理程序,能够抢占和挂起Ring 0级别的代码。Ring -2(系统管理模式,SMM )代码可以进一步抢占Ring -1级别的代码,同时可以自由访问平台硬件,并且在许多情况下还能够绕过可信任执行技术(Trusted Execution Technology),所以,它是现代x86处理器中权限最高的运行级别。考虑到SMM极有可能被滥用,所以我们需要借助数不清的安全机制来包含它。然而,由于该体系结构极其复杂,所以简单实现权限隔离是完全不可能做到的,因此,通过精心利用体系结构中那些匪夷所思的特性,是完全能够绕过SMM的安全保护的。

二、SMM安全措施概述

系统管理模式安全模型的基础,是以系统管理RAM(SMRAM)这个内存区域自身安全并且保护措施良好为前提条件的。从概念上讲,SMM代码只能存在于SMRAM之中,并且SMRAM中的内容,只有当处理器进入SMM模式之后才能进行访问。这样的话,SMM代码能够访问系统上的任何其他东西,但是系统上的任何其他东西都不能访问SMM代码。内存控制器集线器(MCH)介于处理器芯片和内存之间,它实现了SMRAM的隔离性。如果处理器没有进入SMM模式,则MCH就会阻止任何对于SMRAM的访问。如果处理器进入SMM模式,MCH则允许访问它。

三、安全违规

X86的本地高级可编程中断控制器(LAPIC,以下简称APIC)负责处理发送给处理器的中断事件。APIC原本是一个独立的电路,后来被集成到P5微架构的处理器芯片上了。为了在管理APIC时提高访问速度和灵活性,该芯片的寄存器被映射到处理器的一段4KB内存中,其地址为0xFEE00000到0xFEE01000。但是,如果某软件已经使用这段内存用于其他目的,这时就会引起冲突。为了解决此问题,P6系列处理器对APIC进行了扩展,允许它将寄存器重新映射到内存中的其他区域。然而,这修正的只是遗留系统中的一个罕见的问题,并且对于现代系统来说,既不不是必需的也不会用到;但是,现代的处理器仍然支持APIC的重映射功能。

谁曾想到,就是这种古老的重定位APIC寄存器的功能,却在一个完全不相关的处理器体系结构组件(即系统管理模式)中引起了复杂的安全漏洞。如果APIC寄存器窗口移动到了SMRAM范围中,那么本应发送给MCH进行裁决的内存访问,却被APIC给提前截取了,这样MCH就再也收不到了。这样一来,ring 0代码就能直接影响SMM了,从而导致两个运行模式之间基本的架构隔离特性也就荡然无存了。

四、内存水坑攻击

APIC寄存器窗口的固定大小是4KB,并且对4KB的边界对齐。处理器这对于这些APIC寄存器具有有限的控制,但是在4KB寄存器窗口中大部分都被硬编码为0。实际上,这就使得ring 0程序代码可以通过重定位APIC而将内存访问引流到单个SMRAM页面上——对于该内存页面的读操作,返回0,对于该页面的写操作,则直接丢弃。考虑到APIC地位特殊,并且APIC数据也难以控制,所以这个漏洞利用起来极其困难,但是,却也不是无法利用。

五、SMM处理程序

为了将运行权限从ring 0提升到更高级别的ring -2,了解一下可以用于攻击向量的SMM代码还是很有帮助的。SMM代码是在自举过程中,由系统固件进行安装的,这使得它很难成为一种广泛的攻击目标。然而,系统固件的选择组件取自Intel提供的一组UEFI模板代码。这些代码就是原始SMM的入口点,并且,现代系统几乎都部署了这些模板代码。因此,针对这个特定的代码串的攻击,其覆盖面是最广的。SMM入口点的模板代码如图1所示。

        8000 mov  bx, off:unk_8091  ; load offset to GDT descriptor
        8003 mov  eax, cs:0FB30h    ; load physical address of GDT 
        8008 mov  edx, eax 
        800B mov  ebp, eax 
        800E add  edx, 50h ; 'P' 
        8012 mov  [eax+42h], dx     ; initialize segments in GDT 
        8016 shr  edx, 10h 
        801A mov  [eax+44h], dl
        801E mov  [eax+47h], dh
        8022 mov  ax, cs:0FB38h     ; load expected size of the GDT 
        8026 dec  ax                ; decrement total size
        8027 mov  cs:[bx], ax       ; save size to the GDT descriptor
        802A mov  eax, cs:0FB30h    ; reload GDT base address
        802F mov  cs:[bx+2], eax    ; save base address to descriptor
        8034 db   66h 
        8034 lgdt fword ptr cs:[bx] ; load new GDT 
        8039 mov  eax, 0
        803F mov  cr3, eax 
        8042 mov  eax, 668h
        8048 mov  cr4, eax 
        804B mov  ax, cs:0FB0Eh     ; load expected lmode cs selector
        804F mov  cs:[bx+48h], ax   ; patch selector into lmode jmpf 
        8053 mov  ax, 10h           ; load hardcoded pmode cs selector
        8056 mov  cs:[bx-2], ax;    ; patch selector into pmode jmpf 
        805A mov  edi, cs:0FEF8h    ; load smbase
        8060 lea  eax, [edi+80DBh]  ; compute offset of insn at 80db
        8068 mov  cs:[bx+44h], eax  ; patch offset into lmode jmpf 
        806D lea  eax, [edi+8097h]  ; compute offset of insn at 8097
        8075 mov  cs:[bx-6], eax    ; patch offset into pmode jmpf 
        807A mov  ecx, 0C0000080h
        8080 mov  ebx, 100011b
        8086 mov  cr0, ebx          ; switch to 16 bit pmode
        8089 jmp  large far ptr 0:0 ; switch to 32 bit pmode

图1 在现代系统上广泛使用的SMM入口点的模板代码

对于上面的SMM模板代码,存在2到3个常见的变体,这说明过去几十年来SMM入口点已经发生了一两次变动,这就是说,攻击代码还需适应其他版本。不过根据我们的研究发现,上面的SMM处理程序入口是最为常见的一种。

六、提权

该SMM处理程序模板代码是正确的,并且也能够防御普通攻击向量,但可能是属于巧合的缘故,它特别容易受到内存水坑攻击。重要的是,该处理程序会从一个位于SMM中的0xFB00处的数据结构中加载关键数据。为此,它要用到上面所示的8003、8022、802A、804B和805A等处的指令。由于这个结构位于SMRAM,因此,ring 0级别的代码是无法通过正常的手段对其进行访问或者修改的。然而,在陷入系统管理中断之前,如果重映射本地APIC到这个结构所在内存并将其覆盖的话,这些指令就会通过APIC访问寄存器中的内容,而非从内存获取数据。由于APIC中的这些寄存器被硬编码为0,并且不能修改,这意味着8003、8022、802A、804B以及805A的读操作是可以被配置的,即从SMM之外读取,并且只读取0。这实际上就是允许我们对SMM进行极其有限的控制,但是如果设计的巧妙的话,还是能够利用SMM代码生成一个畸形GDT和和跳转指令,从而将控制权转移到SMRAM之外,从而实现利用恶意ring 0代码劫持SMM了。

在利用APIC覆盖0xFB00处的SMM结构之后,继续:在8003处的指令按照GDT的基址读取0;后续指令在物理地址0附近一个不存在的GDT中设立GDT描述符;8022处的指令从内存载入该GDT的大小,由于被APIC覆盖,所以它的大小被误认为是0;8026处的指令将GDT的大小重新计算为0xFFFF,而8027处的指令则讲这个大小保存到一个GDT描述符中;802a处的指令则将处于攻击者控制之下的一个SMRAM之外的地址作为GDT基址加载进来。这样,当新的段选择器被载入SMM 时,ring 0级别的恶意代码就可以控制内存布局了,这是在8089处发生跳转时发生的。当8075处的自修改代码执行并且从水坑的805A处取出数据后,最后一条指令就会被弄成“jmp far ptr  0x10:0x8097”了。通过在地址0处放置一个精心设计的GDT,并令0x10处的描述符指向SMRAM外部的地址,那么0x8097处的有效荷载就能拦截SMM的执行流程,并获得SMM相应的权限。

七、有效攻击载荷

在图2中,我们提供了一个攻击原型代码,该代码已经在选择的某些处理器模式上面验证通过。

        TARGET_SMBASE      equ 0x1f5ef800
        GDT_ADDRESS        equ 0x10000
        FJMP_OFFSET        equ 0x8097
        DSC_OFFSET         equ 0xfb00
        DESCRIPTOR_ADDRESS equ 0x10
        APIC_BASE_MSR      equ 0x1b
        SINKHOLE           equ ((TARGET_SMBASE+DSC_OFFSET)&0xfffff000)
        PAYLOAD_OFFSET     equ 0x1000
        CS_BASE            equ (PAYLOAD_OFFSET-FJMP_OFFSET)
        APIC_BSP           equ 0x100
        APIC_ACTIVE        equ 0x800
        wbinvd
        mov dword [dword GDT_ADDRESS+DESCRIPTOR_ADDRESS+4],
            (CS_BASE&0xff000000)|(0x00cf9a00)|(CS_BASE&0x00ff0000)>>16
        mov dword [dword GDT_ADDRESS+DESCRIPTOR_ADDRESS+0],
            (CS_BASE&0x0000ffff)<<16|0xffff
        mov eax, SINKHOLE | APIC_ACTIVE | APIC_BSP
        mov edx, 0
        mov ecx, APIC_BASE_MSR
        wrmsr
        jmp $

图2 水坑攻击原型

该攻击是由运行在ring 0级别的内核驱动代码发起的,并假设使用一致的内存映射技术。该示例代码的目标为BSP CPU内核,且系统的SMBASE位于0x1F5EF800。这里的GDT地址是根据APIC寄存器在GDT描述符创建时读取的内容进行调整的。该攻击将SMM执行流程因导至SMRAM之外的一个辅助有效载荷,其位于PAYLOAD_OFFSET。这个辅助有效载荷是以ring 0权限进行安装的,在通过水坑攻击劫持SMM处理程序之后,这个有效载荷便具有S了MM级别的运行权限。至于辅助有效载荷的具体作用则留给读者去想象,但通常用途包括高度潜伏的持久性rootkit、修改硬件以及破坏系统等。
