MAC OSX未授权跨应用资源访问

2015-06-19 10:49:07 阅读：8855次 点赞(0) 收藏 来源： 360安全播报

今天的操作系统多数将应用与它们的资源隔离 ，通过一种应用隔离机制：每一应用被限制在它的区域内（拥有最小特权），且需要其它行为（访问照相机，录音机等），这些行为来自系统或用户。这种机制称为沙盒，它能防止程序被恶意破坏。 这种安全模型已被多数主流的操作系统（包括Windows，MAC OS X，android，IOS等）采用 。

造成这种问题是由于威胁模型的缓解方案和传统的失误（系统设计师和应用开发者很少关注于应用的权限问题，当一些应用属于相同的用户时，应该作为不信任对象对待它们，并对应用到应用和应用到系统之间的交互进行适当的安全检查。

恶意代码在一些隔离机制管理下访问其他应用的资源（合法应用使用的），或在它未授权时也这么做。我们称这种安全威胁为未授权跨应用资源访问（unauthorized cross-app resource access）或XARA。

该问题可能会造成Password Stealing，Container Cracking，IPC Interception和Scheme Hijacking。

一些安全缺陷造成的，如Scheme Hijacking，只有系统级的安全解决方案才能修复它们。

弱点应用的检测

为了更好的理解安全缺陷，我们开发了Xavus, 一种 XARA漏洞扫描器（静态审查Apple应用的二进制来识别那些XARA威胁的影响）。

缓解方案

定位XARA问题比重现它更难。

OS本身经常不知如何保护第三方资源的保护方案。

这里可能需要app的开发者给出适当的接口来将它们指定并使用它们的分离策略。一个显著的范例是keychain，因为OS没有条件决定解析某一应用使用某个项目。

由于如此复杂，在Apple找出一种方法让开发者修复它们之前，这些安全缺陷将可能停留一段时间，

确实，在10月份我们已报告了Apple的keychain问题，不久，Apple使用一种随机用户名来patch它的一些应用，它将变得无效。

找出一种long-term是一种挑战，对于一些缓解威胁来说很重要。作为一运行在用户态的第三方应用，部署简单Xavus扫描器可以简单的保护一些Apple的用户。

想法与实现

我们的XARA扫描器用于审查公共信息（不管系统是否发生改变）来检测是被劫持的某应用是服务，资源还是channel

正如我们之后将要展示的，这种设计可让扫描器正常有效地工作。

我们的应用注册文件系统事件（用API FSEvents），当特定的文件被修改时，它是一种问题。扫描器在/Library/Keychains/ 和∼/Library/Keychains/下监控keychain文件.

典型地，某一系统app不共享第三方应用的列表。一旦被检测，扫描器将通知用户存在签字的风险

当谈到第三方应用时，我们可以做的是以离线分析的方式为流行的MAC应用创建profiles。每个profile含有ACL（应用应该使用它），它与某一profile（从keychain到尝试检测某一利用被解析）进行比较。

对于Scheme和BID，我们的扫描器凭借事件 API FSEvents保持对新安装的应用的跟踪，扫描器通过它的plist运行来找出注册URL scheme还是它的helper程序或XPPC服务与系统的冲突事件。检测某一利用时发生冲突，或来自新的应用或存在的应用，因此触发警告。

在Android上的XARA攻击案例

事实上，在android上先占另一应用的scheme比较困难，无论其中两个应用有没有注册scheme，Android总是通知用户并让他们做决定。同时Facebook应用也修复过相应的问题，scheme劫持是他们并不关心的安全问题。

在Apple平台上的安全问题

之前有学者研究过在IOS APIs多种绕过安全检查的安全问题及他们让恶意软件传播的用法。理解Apple内部应用交互的安全冲突且其之前从未完成过沙盒的设计。Fireye曾经发现过IOS scheme的劫持风险，其展示了简单的钓鱼范例。

总结

通过以上描述：app到app及app到system之间交互的问题，检测技术及某一威胁的缓解方案，更好地理解了当今存在的安全问题。未来系统中应该使用更有效的app隔离机制来避开这些安全问题。
