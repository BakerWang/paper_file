Xss的简单发掘

2014-12-15 11:24:49 阅读：12260次 点赞(0) 收藏 来源： J4s0n@AptSec Team

在前面的内容中，我向大家介绍了一些XSS跨站脚本的基础概念，相信大家现在对跨站脚本有了一定的认识。接下来，我们将讲述如何去发掘一些简单的XSS漏洞。

下面演示的示例可能比较简单，如想了解更多发掘和测试XSS的技术，请关注APTSec官方博客。

  1. 搭建测试环境

在发掘xss之前，首先要在本地搭建一个Web服务器，简便起见，可以使用AspServer。

AspServer是一款绿色的ASP服务器，使用它可以便利地开发和调试ASP程序，其下载和安装过程比较简单，这里不再叙述了。

待环境配置完毕，还需要下载一套用来测试的Web程序，这里以“ok3w新闻发布系统”为例。程序下载完后，把相关的程序文件压缩到网站根目录，然后启动AspServer，最后用浏览器打开http://localhost或者http://127.0.0.1即可访问该站点，如图1所示。

现在，我们要开始对“***新闻发布系统”进行XSS漏洞发掘和测试。进行安全测试的方法有几种，这里使用灰盒测试。什么是灰盒测试？要了解此概念 首先需要理解黑盒测试和白盒测试，是指知道程序内部的结构，也就是获取源代码的情况下对软件进行测试；黑盒则完全相反；而灰盒测试介于两者之间。

  1. 发掘反射型的XSS

众所周知，数据交互（即输入/输出）的地方最容易产生跨站脚本，因此，我们可以着重对网站的输入框、URL参数处进行测试。当然，所有来自COOKIE、POST表单、HTTP头的内容都可能产生XSS。

浏览***新闻发布系统的页面，可以看见右上方有个“站内查找”的功能，如图2所示。在搜索框中输入跨站测试代码XSS<XSS>’”，然后单击【提交】按钮，提交后注意观察页面的返回的结果，如图3所示。

“XSS’ ‘ “” 搜索结果>>

图1 ***新闻发布系统主页

图2 输入XSS代码

图3 搜索返回页面

从图3中可以看到，我们输入的XSS测试代码中的<XSS>没有显示出来，这种情况下，测试代码可能已被程序过滤，也可能程序接受了但 没有显示出来，因为在HTML语言中，<>表示HTML标记，不会被浏览器直接显示出来，只有通过查看相应的源文件才能看到。所以，我们此时 需要查看页面的源文件，然后搜索关键字XSS，这时候会发现如图4所示的代码片段：

可以看到，我们输入的<XSS>确实被完整地写进了页面，由此可证明***新闻发布系统没有过滤<>等关键字符。那么我们就可以直接注入一个完整的<scipt>标签，从而产生一个XSS。

重新回到搜索栏，输入完整的XSS代码<script>alert(/xss/)</script>，然后单击【搜索】按钮，随后在其浏览器界面上弹出了一个内容为XSS的对话框，如图5所示。

图4 查看XSS后的HTML源文件

图5 出现的XSS漏洞

细心的读者可能已经发现，此时浏览器地址栏上的链接变成了：

http://192.168.1.100/search.asp?keyword=%3Cscript%3Ealert%28%2Fxss%2F%29%3C%2Fscript%3E&Submit=%CB%D1%CB%F7

因此，我们可以证实***新闻发布系统存在一个反射型的XSS。

我们利用黑盒技术发掘到跨站脚本漏洞后，还要利用白盒技术分析一下该漏洞的成因。找出相关程序文件search.asp，发现有如下代码：

<%

Set Page = New TurnPage

keyword=CmdSafeLikeSqlStr(Request.QueryString(“keyword”))

If keyword=”” Then

Call MessageBox(“请输入需要查询的关键词。”,””)

End If

%>

上述代码中，keyword变量代表输入的查询内容，然后程序使用Request.QueryString()直接获取该变量，随后只经过 CmdSafeLikeSqlStr()函数的处理。CmdSafeLikeSqlStr()在APPCode/fun/function.asp中有定 义，找到该文件可以了解它的作用：

Function CmdSafeLikeSqlStr(Str)

Str = Replace(Str,”‘”,”””)

Str = Replace(Str,”[“,”[[]”)

Str = Replace(Str,”%”,”[%]”)

Str = Replace(Str,”_”,”[_]”)

CmdSafeLikeSqlStr = Str

End Function

以上代码的功能是把查询变量中含有的’ [ % _之类的字符替换成” [[] [%] [_]。显然，这里忽略了对<>等关键字符的过滤，导致用户可任意注入HTML标记来引发XSS。

  1. 发掘持久型的XSS

通过分析源程序代码，我们对漏洞成因有了更好的了解，但是在发掘XSS之前，黑盒测试的效果显然比白盒测试的好，而结合利用这两种方式的灰盒测试技术，能大大提高我们发掘XSS的效率。

接下来继续测试“***新闻发布系统”是否还存在其他跨站漏洞。浏览该系统的其他也没，发现有一个“留言交流”的功能，页面地址为http://192.168.1.100/guest.asp。在这里用户可以任意输入留言，如果程序没有慎重过滤这些留言信息，很可能会产生一个持久型或存储型的XSS漏洞。

如图6所示，在留言页面的留言处输入如下内容：

<script>alert(/xss/)</script>

图6 在留言内容处输入XSS语句

单击【提交】按钮后，发现页面没有任何迹象。先不要着急，因为用户提交的留言一般要通过后台管理员的审核才会显示出来，***新闻发布系统亦是如此。所以在提交XSS代码后，我们以管理员身份登录后台查看留言，后台管理的登录地址为：

http://192.168.1.100/admin/ad_login.asp

打开该页面，输入默认管理员账号和密码admin，如图7所示，然后登录。

图7 登录后台

成功登录后台后，单击“留言/评论管理”栏的“等审留言”链接，这时页面显示的内容如图8所示。

图8 后台查看留言

从上图中可以看出，我们输入的XSS语句并没有顺利执行，反而被浏览器直接显示出来。出现这种情况，极有可能是程序对这条XSS代码进行了转义，查看一下当前页面的源代码，如图9所示。

图9 留言页面的源文件

经过一轮分析后发现，输入的XSS代码不仅没有被过滤，而且已经成功插入到网页内容中，之所以没有执行代码，是因为XSS代码被嵌入 到<textarea>标签之中，然后被浏览器直接显示出来。因此，在构造XSS代码时，需要闭合<textarea>标签，重 新修改XSS代码：

</textarea><script>alert(/xss/)</script><textarea>

这两个<textarea>分别用来闭合前后的<textarea>标记。提交此代码，然后再以管理员身份查看留言，这时候页面成功弹出了带有XSS内容的对话框，如图10所示。

图10 成功弹出XSS

以上就是一个典型的持久型XSS，并且这个XSS在管理员查看留言时才触发。这种XSS场景可以发挥的空间很大，因为它攻击的对象是后台管理员，攻 击者能利用此种XSS劫持管理员会话而执行任意操作，比如修改密码、添加新闻、备份数据等（攻击方法：调用XSS平台，Xss平台网 址：http://xssec.cc/！）

最后，分析漏洞成因，找到处理留言的程序文件***3w_Guest.asp，相关代码如下：

‘接收表单

Private Sub GetFormData()

UserName = Trim(Request.Form(“username”))

Mail = Trim(Request.Form(“email”))

Homepage = Trim(Request.Form(“homeurl”))

Content = Trim(Request.Form(“content”))

QQ = Trim(Request.Form(“qq”))

TypeID = Trim(Request.Form(“TypeID”))

If TypeID=”” Then TypeID=1

TableID = Trim(Request.Form(“TableID”))

If TableID=”” Then TableID=0

AddTime = Now()

Ip = Request.ServerVariables(“REMOTE_ADDR”)

IsPass = 0

Ad_ask = “”

End Sub

‘更新记录集

Private Sub UpdateRs(ByRef Rs)

Rs(“UserName”) = UserName

Rs(“Mail”) = Mail

Rs(“Homepage”) = Homepage

Rs(“Content”) = Content

Rs(“QQ”) = QQ

Rs(“TypeID”) = TypeID

Rs(“TableID”) = TableID

Rs(“AddTime”) = AddTime

Rs(“Ip”) = Ip

Rs(“IsPass”) = IsPass

Rs(“Ad_ask”) = Ad_ask

End Sub

上述代码仅用了Request.Form()函数来获取表单数据，包括用户名、邮箱地址、留言内容等信息，然后简单地去除两边的空格，就直接把留言信息写进数据库中，程序没有过滤任何有害字符，从而导致一个持久型XSS的产生。

可见，XSS跨站脚本始终是Web应用程序中最容易产生、最容易被忽略的安全漏洞。

最后再来个小科普：如何利用xss来进行APT攻击

问：Xss可以被利用来做什么？

答：XSS技术的运用方式灵活多变，利用它可以劫持浏览器用户的会话、窃取客服端的Cookies、网络钓鱼等，在特定场景下或配合其他漏洞，威力会更大！

那么，我向大家分享一个思路。编写一个JS病毒脚本，其脚本作用是强制远程控制受害者下载木马并且运行该木马，最后再利用XSS技巧来实现这一脚本 的运行。（劫持用户浏览器会话）受害者的计算机运行了木马后，我们便可以对他的计算机为所欲为了。所以我们可以把这个看成一种APT攻击手法！当然实现这 个思路是不容易的，你首先要考虑到受害者的浏览器是否防脚本注入，如果是，那么该如何绕过？这是一个值得我们思考的问题。还有对于编写这样的一个脚本，小 编对于JS脚本的熟练程度还不是很好，编写起来会很困难，但小编会尽快编写好的，并将此脚本添加至XSS平台的模块。

文章到此结束，如有对这个思路有兴趣的、愿意一起研究的，请联系E-mail:admin@xssec.cc。
