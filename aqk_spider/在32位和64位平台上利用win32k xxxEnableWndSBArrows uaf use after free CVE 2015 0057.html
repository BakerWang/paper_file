在32位和64位平台上利用win32k!xxxEnableWndSBArrows uaf（use-after-free）(CVE 2015-0057)

2015-07-14 10:17:07 阅读：12747次 点赞(0) 收藏 来源： 360安全播报

介绍

win32k.sys中存在uaf (CVE-2015-0057)漏洞并且可以开发32位和64位平台下使用的利用程序。影响 XP 至Windows 8.1 (带异常).这篇 writeup 描述了在两种不同平台上实现利用的方法，它们在细节上稍有不同。实现该bug的利用相当困难。我很希望能够了解到更简单的实现方式。

存在的bug

反汇编win32k!xxxEnableWndSBArrows的情况如下：

未安装补丁

.text:FFFFF97FFF1B157D mov r8d, r13d .text:FFFFF97FFF1B1580 mov rdx, r14 .text:FFFFF97FFF1B1583 call xxxDrawScrollBar ; xxxDrawScrollBar could've issued usermode callback .text:FFFFF97FFF1B1588 jmp short loc_FFFFF97FFF1B1519 [...] .text:FFFFF97FFF1B1519 mov eax, [rbx] ; Dereference tagSBINFO ptr without sanity check .text:FFFFF97FFF1B151B mov ebp, 0FFFFFFFBh .text:FFFFF97FFF1B1520 xor eax, esi

如上代码，win32k!xxxDrawScrollBar可在当前情况下调用用户态的tagSBINFO（由攻击者释放）。当返回如上代码时，位于0xFFFFF97FFF1B1519的指令将使用非法指针。

安装过补丁

.text:FFFFF97FFF1D69C3 xor r8d, r8d .text:FFFFF97FFF1D69C6 mov rdx, rbp .text:FFFFF97FFF1D69C9 call xxxDrawScrollBar ; Could've issued usermode callback
.text:FFFFF97FFF1D69CE cmp rbx, [rdi+0B0h] ; Is tagSBINFO ptr still correct? .text:FFFFF97FFF1D69D5 jz short loc_FFFFF97FFF1D69E4 ; If so, continue .text:FFFFF97FFF1D69D7 .text:FFFFF97FFF1D69D7 loc_FFFFF97FFF1D69D7: .text:FFFFF97FFF1D69D7 mov rcx, rbp .text:FFFFF97FFF1D69DA call _ReleaseDC .text:FFFFF97FFF1D69DF jmp loc_FFFFF97FFF1D6958 ;Jump to exit function block .text:FFFFF97FFF1D69E4 ; --------------------------------------------- ------------------------------ .text:FFFFF97FFF1D69E4 .text:FFFFF97FFF1D69E4 loc_FFFFF97FFF1D69E4: .text:FFFFF97FFF1D69E4 mov eax, [rbx] ; Safely deref the unchanged tagSBINFO ptr .text:FFFFF97FFF1D69E6 xor eax, r14d

如上为打过补丁的版本，我们看到在被解引用前，会对原始tagSBINFO指针做null检查。之后会提供更多相关结构的信息。

触发bug的关键

关键在于拦截通过win32k!xxxDrawScrollBar()触发的用户态回调。因此我们可以在win32k!xxxEnableWndSBArrows()继续执行前改变堆块的内容。

桌面堆

Win32k.sys使用桌面堆来存储GUI对象（与给出的desktop相关联）。这包括窗口对象以及它们的相关结构，如属性链表，窗口文本，和滚动条。

监控桌面堆的分配情况

使用如下WinDbg脚本

64位平台下用于监控堆的脚本

ba e 1 nt!RtlFreeHeap ".printf\"RtlFreeHeap(%p, 0x%x, %p)\", @rcx, @edx, @r8; .echo ; gc"; ba e 1 nt!RtlAllocateHeap "r @$t2 = @r8; r @$t3 = @rcx; gu; .printf \"RtlAllocateHeap(%p, 0x%x):\", @$t3, @$t2; r @rax; gc";

32位平台下用于监控堆的脚本

ba e 1 nt!RtlAllocateHeap "r @$t2 = poi(@esp+c); r @$t3 = poi(@esp+4); gu; .printf \"RtlAllocateHeap(%p, 0x%x):\", @$t3, @$t2; r @eax; gc"; ba e 1 nt!RtlFreeHeap ".printf\"RtlFreeHeap(%p, 0x%x, %p)\", poi(@esp+4), poi(@esp+8), poi(@esp+c); .echo ; gc"

除了这些断点脚本外，因为桌面堆只是在用户态中后端分配器的一种简化形式，我们可以在WinDBG中补充使用!heap命令来进一步观察。

填满堆holes

在多数分配桌面堆的情况相当有趣并且其直接与窗口对象相关联，通过tagWND结构来管理它，这意味着如果我们想要分配一个任意大小的堆块，那么我们首先需要一个被分配的窗体对象来进行相关的交互操作。

验证 feng shui 布局

有趣的事是，桌面堆将作为只读的结构被映射进用户态中，PEB含有一个未公开的结构，其被称作Win32ClientInfo。大概被定义为如下所示：

typedef struct _CLIENTINFO { ULONG_PTR CI_flags; ULONG_PTR cSpins; DWORD dwExpWinVer; DWORD dwCompatFlags; DWORD dwCompatFlags2; DWORD dwTIFlags; PDESKTOPINFO pDeskInfo; ULONG_PTR ulClientDelta; // incomplete. see reactos } CLIENTINFO, *PCLIENTINFO;

首先PDESKTOPINFO结构含有如下信息：

typedef struct _DESKTOPINFO { PVOID pvDesktopBase; PVOID pvDesktopLimit; // incomplete. see reactos } DESKTOPINFO, *PDESKTOPINFO;

在windows 7 及之后发行的平台上，已经导出了gShared地址（存储于user32.dll）。该地址可以很容易地被找出。在多数系统中定义的结构如下：

kd> dt !tagSHAREDINFO win32k!tagSHAREDINFO +0x000 psi : Ptr32 tagSERVERINFO +0x004 aheList : Ptr32 _HANDLEENTRY +0x008 HeEntrySize : Uint4B +0x00c pDispInfo : Ptr32 tagDISPLAYINFO +0x010 ulSharedDelta : Uint4B +0x014 awmControl : [31] _WNDMSG +0x10c DefWindowMsgs : _WNDMSG +0x114 DefWindowSpecMsgs : _WNDMSG

实现命令执行

因此如果我指明泄露/读取X或覆写Y，那么可以用更早前的腐烂stage中已经进行过的操作来完成。我们还准备覆写函数指针并且将其重定向到payload区域。最为常见的实现方法就是覆写nt!HalDispatchTable中的第二个条目，它通常含有HalQuerySystemInformation()函数。接着你可以从用户态中强制调用NtQuery InternalProfile() 函数的指针。我们需要泄露内核地址来找到nt!HalDispatchTable。为了实现该目的，我使用标准的NtQuerySystemInformation()函数从你抽取的基地址中取回模块信息。

  // 11 corresponds to SystemModuleInformation class, which is
undocumented...
  rc = NtQuerySystemInformation((SYSTEM_INFORMATION_CLASS)11,
pModuleInfo, 0x100000, NULL);

接着我打开本地的ntoskrnl.exe副本并查找nt!HalDispatchTable的偏移, 然后我可以更简便地应用被泄露的基地址。我使用read操作来读取原HaliQuerySystemInformation()的地址（没有被导出）以便在之后可被修复，并且之后使用write操作来腐烂带有shellcode地址的函数指针（位于用户空间或内核空间；之后会提到更多相关信息）。

绕过SMEP

窗体对象处理并调用GetInternalWindowText()。因此问题在于我们正使用的指向内存映射区域（含有我们的shellcode）的页表条目中窗体结构的strName成员，它在我们的进程页表中属性为private。当dwm.exe要求内核给它命名时，将出现错误的页表条目并且内核将会了解到strName.Buffer的地址不为NULL，接着解引用地址（将会是 非法的）。这将导致机器BSOD。

简要步骤如下:

1.  创建一段窗体文本的buffer(含有stage one的payload)并计算内核地址。

2.  使用自映射技巧从上一步中计算出内核地址的页表条目。

3.  使用原始的写操作来设置页表入口的执行位。

4.  覆写nt!HalDispatchTable

5.  调用NtQueryInternalProfile()跳转到payload。

6.  用cr4寄存器关闭SMEP保护机制并跳转到stage two的用户态payload。

7.  执行stage two中的用户态payload来提权并返回。

8.  用cr4寄存器重置SMEP以防patchguard 会complain并完美返回。
