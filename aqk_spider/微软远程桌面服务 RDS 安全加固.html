微软远程桌面服务（RDS）安全加固

2015-05-26 07:38:18 阅读：25450次 点赞(0) 收藏 来源： 360安全播报

作为系统管理员，我们通常需要采用不同的对策来降低那些我们不能完全解决的问题所带来的风险。这篇文章旨在向大家介绍如何降低由网络上的远程桌面服务(之前称之为终端服务)所引起的风险。

我将要披露的风险包括：

1. 中间人攻击

2. 嗅探/流量捕获

3. 暴力破解攻击

4. 信息披露

威胁

中间人(MITM)攻击

在这里，最基本的一个前提就是一名攻击者可以通过一系列方法使得RDP的流量数据经过他所控制的主机。这样就攻击者就能够查看这些流量信息［1］了，而且在某些情况下还可以操作这些数据，并降低服务器端与客户端通信链路的安全等级。

嗅探/流量捕获

为了查看流量数据信息，攻击者并不需要真正参与客户端与服务器端的通讯。因为有大量的技术能够使攻击者去记录网络中客户端与服务器端通信的网络流量信息。即便这些流量数据使用了TLS协议进行加密，我们也可以利用TLS验证漏洞来对捕获到的数据包进行离线攻击［2］。

暴力破解攻击

现在有很多的工具能够尝试各种用户名与密码的组合，这些工具的目的是为了确定有效的证书信息，例如Hydra［3］和ncrack。这种攻击的速度非常的快，而且效率也很高。但这种攻击方法也有它的缺点，即有可能导致系统拒绝服务。这是由于目标RDS服务器在用户登录之前就已经向用户分配好了相应的资源所导致的。一个有设计缺陷的暴力破解攻击能够通过同时尝试进行多个链接而使目标主机的CPU占用率达到100%，但这种现象并不常见。

信息泄露

服务器端未经过身份验证的RDP链接会暴露目标主机的敏感信息。用户名，域名以及网络中其他主机等黑客感兴趣的信息会在连接成功之后显示出来。右下方的屏幕截图显示，用户以管理员的身份进行登录并链接至WinServer01。管理服务器和工作站通常存储有大量的证书信息，而且通常绕过这些防火墙的方法比绕过其他大多数主机防火墙的方法要多得多。

降低风险

好消息是，可以通过更改3个设置选项来降低这些风险。微软操作系统以及许多第三方RDP客户端软件都兼容和支持这些设置的修改。并且这些设置可以通过GUI，PowerShell和组策略进行修改。

为安全层设置SSL(TLS 1.0)

这个设置需要使用TLS 1.0或更高版本的加密来保护会话的安全，而不是使用传统的RDP加密。为了增加加密算法的强度，还可以通过服务器端的TLS(x.509)验证来启用中间人攻击检测。最理想的情况就是每一个服务器都拥有一个由授信任的机构所颁发的证书，但即便是使用自签名证书，也可以让细心的用户在首次链接之后便能够对中间人攻击进行检测 。如果客户端与服务器端都支持并且使用了TLS加密套件，那么即便是得到了RDP服务器端的TLS证书信息，也无法对嗅探到的RDP会话信息进行解密。除此之外，任何在加固服务器端或客户端TLS设置上所付出的努力都将会使它们的RDP会话更加的安全。

在设置完全正确的环境中，这种设置能够大幅降低中间人攻击和嗅探的风险。这样就能够向股权持有者以及利益相关的第三方保证他们的通讯安全，并且加密所使用的加密算法是众所周知的并且被广泛接受的。

启用网络级身份验证(NLA)

网络级身份验证需要用户在与服务器端建立会话之前，通过RDP去验证一个链接。我们可以利用Kerberos,NTLM以及PKI来进行验证。除此之外，由于它使用的是微软的CredSSP协议，会话期间所有的流量信息都将通过TLS 1.0或更高版本的协议进行发送。

使用网络级身份验证能够大幅降低信息披露所带来的上述风险。NLA功能在Windows 2012 R2中是默认启用的。

设置高等级的加密

Windows操作系统默认允许服务器端与客户端就加密等级进行协商。为了将加密等级设置为”高”，将需要至少128位加密密钥，否则服务器端将不会允许客户端进行连接。基于这种环境的需求，加密等级可以设置为“FIPS”，但这种设置方案并不能够直接解决上述的问题。

部署

可以通过各种方式将这些设置部署至环境中。

GUI

在Windows 2008和2008 R2中，设置的值可以通过GUI进行修改，点击开始－管理工具－远程桌面服务，然后点击“远程桌面会话设置”。右键点击RDP－tcp然后点击“属性”。上面所描述的所有设置都可以在常规设置选项中找到。当所有设置都设好之后，点击“应用”。然后这些设置将会立即生效，并且不会影响但前正在进行的会话链接。

PowerShell

下面的一小段PowerShell代码将会对上述三个设置进行配置。它需要管理员权限，并且能够在Windows 2008 R2以及2012 R2中生效。

# Remote Desktop Services: Enable NLA Requirement (Get-WmiObject -class "Win32_TSGeneralSetting" -Namespace root\cimv2\terminalservices -Filter "TerminalName='RDP-tcp'").UserAuthenticationRequired (Get-WmiObject -class "Win32_TSGeneralSetting" -Namespace root\cimv2\terminalservices -Filter "TerminalName='RDP-tcp'").SetUserAuthenticationRequired(1)   
# Remote Desktop Services: Require 'High' level of encryption (Get-WmiObject -class "Win32_TSGeneralSetting" -Namespace root\cimv2\terminalservices -Filter "TerminalName='RDP-tcp'").SetEncryptionLevel(3)   
# Remote Desktop Services: Set Security Layer to SSL (Get-WmiObject -class "Win32_TSGeneralSetting" -Namespace root\cimv2\terminalservices -Filter "TerminalName='RDP-tcp'").SetSecurityLayer(2)  
###############################################################################

# .SetEncryptionLevel(3) - https://msdn.microsoft.com/en-us/library/aa383800(v=vs.85).aspx 
# .SetSecurityLayer(2) - https://msdn.microsoft.com/en-us/library/aa383801(v=vs.85).aspx


组策略

前面两个选项适合在测试的时候使用。在活动目录环境中进行部署将会使用到组策略。我建议用户为这些设置创建一个GPO，这样他们就可以进行部署了。所有相关的设置都可以在 

Computer Configuration\Policies\AdministrativeTemplates\Windows Components\Remote Desktop Services\Remote Desktop Session Host\Security


下被找到。

潜在的问题

现在仍然存在一些需要我们关注的问题。首先，对于那些仍然需要支持Windows XP平台的客户端来说，以下几个操作是你需要用到的。

1. 将系统升级至SP3。

2. 启用Windows XP SP3[8]中的CredSSP。

3. 安装远程桌面客户端7.0[9]。

其次，你需要注意的是，即便没有修改任何的配置和软件，在任何系统环境下的风险都是动态的。
