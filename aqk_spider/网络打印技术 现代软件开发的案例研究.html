网络打印技术-现代软件开发的案例研究

2015-06-23 15:03:45 阅读：5932次 点赞(0) 收藏 来源： 360安全播报



摘要

在这篇文章中，我描述了一个漏洞的利用链。该利用链中的漏洞是我在CUPS系统中发现的，而CUPS是一个开源的打印套件。首先，我分析了一个CUPS字符串处理的漏洞(CVE-2015-1158)。然后我将会讨论该系统的密钥设计以及导致这个漏洞的成因。之后，我会讨论如何利用这个初始漏洞。接下来，我会描述另一个错误漏洞(CVE-2015-1159)，这个漏洞会使第一个漏洞的影响更加严重。最后，我会对CUPS套件的特定功能和配置选项进行描述。

通过这篇分析文章，我希望能够鼓励关于此漏洞以及漏洞缓解措施的讨论能够更加的透明化，并且以此激励其他的研究人员也能够这样做。

背景

 什么是CUPS？

CUPS是一个基于UNIX操作系统的，模块化的、功能丰富的、开源的打印系统。它将大量的打印文件格式抽象集成在了打印机的硬件上，并以此降低了打印机对于特定驱动程序的依赖性。

CUPS是由苹果公司旗下的产品，并且能够运行在Linux，BSD以及各种类似Mac OS X的操作系统上。并且广泛应用于桌面电脑，笔记本电脑，网络打印服务器以及某些嵌入式系统之中。CUPS的实现参考了IPP/2.0和IPP/2.1,这是LPD/LPR打印的最新替代品。CUPS的source tree是非常庞大而且十分复杂的，但并不是独一无二的，因为其中大约有超过10万行的ANSI C代码。

CUPS内部字符串匹配

CUPS系统中大量使用了字符串来管理属性配置。这些字符串是由’_cupsStrAlloc()’分配的。

所有的字符串保存在一个全局的哈希表内。’_cupsStrAlloc()’首先会检查哈希表中的输入字符串(检查内容)。如果字符串在哈希表中，说明它之前已经分配过了，并且它的引用计数会自动增加，并从全局数组中返回一个现存的字符串指针。如果字符串在全局哈希表中没有找到，那么系统会分配一个新的堆块，然后插入一个单一引用。

简单来说，如果两个完全不同的函数为同样内容的字符串调用了’_cupsStrAlloc()’，那么它们会得到同样的返回值，并共享一个堆块指针。

如果’_cupsStrAlloc()’分配了一个字符串，那么这个字符串还必须由函数’_cupsStrFree()’来将其释放。这个函数会使引用计数自减，如果引用计数没有了，那么它便会从全局哈希表中将这个字符串字符串移除，并且释放堆块。

引用计数递减的问题

IPP属性拆解

 IPP属性是各种键值对。属性名有布尔值，整形，日期，字符串等等类型，并且可以进行分组。属性值可以是多种类型（一个属性名可以与多种值对应）。局部属性的种类有’IPP_TAG_TEXTLANG’和’IPP_TAG_NAMELANG’。在通讯协议中，对这些属性的字符集有特殊的规定，即便是每个属性可以有多个值。

 Cups将属性保存在一个’ipp_attribute_t’|’_ipp_attribute_s’结构体中：

cups/ipp.h:674:
typedef struct _ipp_attribute_s ipp_attribute_t;
cups/ipp.h:789:
struct _ipp_attribute_s     /**** 属性****/
{
 ipp_attribute_t *next;    /* 列表中下一个属性*/
     ipp_tag_t group_tag,    /* 任务/打印机/操作组标记*/
   value_tag;    /* 值是什么类型 */
 char    *name;      /* 属性名称*/
 int   num_values;   /* 值名称 */
 _ipp_value_t  values[1];    /*属性值*/
};

不当的拆解－引用计数递减(CVE-2015-1158)

当本地化的多值属性被释放时，语言字符串的引用计数会递减。而当创建了一个打印任务的时候，属性名为’job-originating-host-name’的属性值将会不止一个。在’add_job()’函数中，cups将会错误地释放‘语言’域中的所有字符串，而并没有使用’ipp_free_values()’函数。

scheduler/ipp.c:1626:

 /*
     * 释放旧的字符串…       ←  即便是旧的字符串也需要释放
     */
     for (i = 0; i < attr->num_values; i ++)
     {
       _cupsStrFree(attr->values[i].string.text);
       attr->values[i].string.text = NULL;
       if (attr->values[i].string.language)           ← 一个属性中所有的值
       {
   _cupsStrFree(attr->values[i].string.language);     ←  释放'language' 字符串
   attr->values[i].string.language = NULL;
       }
           }

利用策略

采用ALC

Cups允许HTTP PUT请求来远程修改系统的主配置文件(‘/admin/conf/supsd.conf’)。这是一个特权操作，是在默认配置文件中受ACL保护的：

cupsd.conf:

# Restrict access to the admin pages...
<Location /admin>
 Order allow,deny
</Location>
# Restrict access to configuration files...
<Location /admin/conf>
 AuthType Default
 Require user @SYSTEM
 Order allow,deny
</Location>

代码中的’Location’指令存储在全局数组’Locations’之中，这是’cupsd_location_t’结构体中的一个数组：

scheduler/auth.h:89:

typedef struct
{
 char      *location;  /* 资源地址 */
 size_t    length;   /* 地址长度 */
 ipp_op_t    op;   /* IPP 操作*/
 int     limit,    /* 请求类型限制 */
     order_type, /* 允许或拒绝 */
     type,   /* 验证类型 */
     level,    /*需要的访问等级 */
     satisfy;  /* 满足条件 */
 cups_array_t    *names,   /* 用户名或组名 */
     *allow,   /* 允许 */
     *deny;    /* 拒绝*/
 http_encryption_t encryption; /* 是否加密*/  ← Very existential :)
} cupsd_location_t;

每个结构体中的‘location’域内都含有’Lacation’配置指令的地址，而且正好成为引用计数的字符串。

将库文件写入磁盘

为了使用’SetEnv’命令来运行代码，我需要将共享库写入硬盘预先分配的磁盘空间之中。Cupsd将POST主体保存在硬盘的’RequestRoot’目录中(Mac Os X系统：'/private/var/spool/cups/',Linux系统：'/var/spool/cups')。完整的文件名称取决于函数’cupsdReadClient()’:

CUPS以及浏览器沙盒

很多浏览器沙盒提供了能够执行HTTP请求的特权管理机制来代表无特权的子进程(潜在不受信任的)。当子进程受到同源策略的限制时，一个被感染的子进程将会请求管理器向本地的’cupsd’发送HTTP POST请求，并利用它打破沙盒的限制。

CUPS可能被用于很多意想不到的地方

桌面端以及网络打印服务器端的CUPS漏洞已经被披露出来，且被广泛讨论了。然而，CUPS还有可能被嵌入至其他很多意想不到的地方，那些人也许从未想过要对这些系统进行修复。
