google projectzero:一个完美的BUG FLASH类型混淆利用

2015-07-27 13:40:07 阅读：6958次 点赞(0) 收藏 来源： 360安全播报

对于一些攻击者来说，重点在于利用程序可成功实现目标。也就是说，当利用程序在已知平台和Flash版本的系统上运行时，该程序应该可以成功实现命令执行。构建该类程序的途径之一是利用高质量的漏洞。这篇文章主要描述该漏洞对应的利用程序，并且其中提出的一些因素可以有效地让利用程序变得更可靠。

Bug描述

在Adobe Flash的Button和MovieClip过滤设定程序，允许用任意其它的过滤程序混淆任意过滤类型）中存在的CVE-2015-3077 是一个类型混淆问题。我将其在早前的2015年二月报告给了Adobe并且在该年的五月份被修复了。由于攻击者可以覆写被使用于初始化一个过滤对象的构造函数而造成了该漏洞的存在，漏洞相关代码描述如下：

         var filter = new flash.filters.BlurFilter();
         object.filters = [filter];
         var e = flash.filters.ConvolutionFilter;
         flash["filters"] = [];
         flash["filters"]["BlurFilter"] = e;
         var f = object.filters;
         var d = f[0];

由于operator[]的使用导致该代码有部分被混淆了，要在Flash CS中进行编译操作有必要这么做。其等价于如下代码（不确保其可以被成功编译）:

         var filter = new flash.filters.BlurFilter();
         object.filters = [filter];
         flash.filters.BlurFilter = flash.filters.ConvolutionFilter;
         var f = object.filters;
         var d = f[0];

该代码设定了对象的过滤区域，Button或BlurFilter对象的MovieClip，然后被Flash本地存储。BlurFilter构造函数会被ConvolutionFilter构造函数覆写。接着该区域过滤程序将得以调用，然而，构造函数已经被覆写，所以此处会调用ConvolutionFilter的构造函数。这将导致通过被返回的BlurFilter来返回ConvolutionFilter对象。

它好像是BlurFilter对象，最终会导致 ConvolutionFilter的区域可被访问，并且当它为任意其它过滤类型时也会遇到类似的情况。对广义数组的控制会有助于利用程序的实现。

如下图解展示了native对象在内存中的布局，在64位linux中使用该漏洞可对native对象进行混淆操作。

AS2 Filter Types

在两种情况下，指针用可被控制的整型或浮点型值进行调整，这意味着指针可以被直接读取或写入。同时，因为对象区域已经被排序并且其大小是基于所定义的类的，它们总位于我们可意料到的区域内，因此读取和写入操作将不会失败。这些性质的存在使利用程序变得更可靠。

利用程序

因为利用该问题将需要多次触发类型混淆问题，开头我已创建了一个进行类型混淆的函数，FilterConfuse.confuse。它也会进行一些清空操作，例如将ActionScipt过滤构造函数往回设置为normal以让存在弱点的函数可以被多次调用，其在该函数外不影响ActionScript的行为。

首先是通过判断出vtable的地址来绕过ASLR。这一实现意味着将会用带有可被控制的重叠成员的vtable对象混淆带有vtable的对象，但是所有过滤对象在相同的偏移上都有vtable，我已经在DisplacementMapFilter中使用了BitMapData对象来判断vtable的地址。

为了判断BitmapData对象在内存中的位置，我已经用BevelFilter混淆了DisplacementMapFilter。这会导致BitmapData指针被存储于DisplacementMapFilter，其被用于调节BevelFilter的颜色属性(shadowColor, shadowAlpha, highlightColor 和 highlightAlpha)。这些属性以两个32位整数的形式返回（上面展示了 scolor下面展示了hcolor），并且当alpha属性访问高8位时，颜色属性将访问每个整数的低24位。在读取这些属性并用位算术操作来将它们连接时，可能会将BitmapData对象的确切地址提取。

检索 BitmapData 指针

接着我们需要在高于BitmapData对象的地址上读取vtable。我已使用了ConvolutionFilter的matrix属性来实现该操作。当属性被设定时，该属性会被存储为一个被分配浮点型数组的指针，当属性被检索到时，将含有这些浮点型的ActionScript数组会被返回。通过将matrix指针设定给BitmapData对象，在内存中可能会作为一个浮点型数组来读出该对象的内容。

为了设定指针，我已经用DisplacementMapFilter对象（和上面使用过的DisplacementMapFilter对象不同）

混淆了ConvolutionFilter对象并将mapPoint属性设定为如上BitmapData对象的指针区域。mapPoint属性用x和y坐标值来描绘，他们都为整型值（如下图所示的p_x和p_y）同时他们也被用于在ConvolutionFilter中对matrix指针进行调节，这样做可以简单地对这个值进行设定。通过读取ConvolutionFilter对象中的matrix数组来读取BitmapData对象中的vtable。（注意：对象已经被混淆为DisplacementBitmapFilter并且在之后会被混淆回ConvolutionFilter来进行该操作）。

检索 vtable 指针值

在这时，由于浮点类型值的使用，让利用程序具有可靠性会变得更加困难。由于那是数组类型，从ConvolutionFilter matrix中会将vtable_low 和vtable_high值读取为浮点型，但不幸的是，不是每个指针的合法值都为合法浮点值。这意味着在对值进行读操作时，会遇到NaN或更糟的情况，不会是相当准确的数值。

该问题的理想解决方案是通过一个将他们解释为整型的getter来获取vtable_high和vtable_low值，但是其中一个操作会失效，由于该限制，过滤区域的成员可能为浮点类型。

幸运的是，AS2虚拟机一般不解释浮点值-当在虚拟机上进行ActionScript的操作时，它会在内存中将某个值转换为浮点类型的值。除了一些特定的操作，例如进行运算操作外，Native操作通常不会造成浮点值被解释。这意味着当matrix数组中的浮点值被复制到vtable_low或vtable_high时，将会在内存中保留它的值，直到该被复制的变量被使用于ActionScript中，或在虚拟机上用native代码进行了运算操作。

因此如果该变量值立即被类型混淆为不同类型的值（支持32位值的所有范围，例如整型，在matrix数组的内存中它将和原始值一样。因此为了避免利用程序中存在的不可靠性因素，在用ActionScript对任意浮点值进行操作前，有必要进行该类型混淆操作。

为了实现该目标，我已经编写了对类型进行转换的类，FloatConverter，其在过滤区域中使用了类型混淆来实现整型到浮点类型和浮点类型到整型的转换。他混淆了ColorMatrixFilter的 matrix属性（没有用ConvolutionFilter的matrix 属性混淆过）一系列的内联浮点值（GlowFilter的 color和alpha属性），它访问不同整型值的字节。

浮点类型转换程序

可成功实现浮点型到整型的转换，但无法很好地完成整型到浮点型的转换。用ActionScript可访问ColorMatrix filter中的color数组，即使是第一个元素被访问，整个数组都被复制。当数组被复制时，每个元素会被转换为一个Number，有时会涉及到访问指针（例如，调用某个对象上的valueOf）。因为颜色数字比整个GlowFilter类还要长，当使用GlowFilter进行混淆时，它会延伸在堆上。这意味着转换程序可以在堆上出现未知数值，如果在被转换为Number时它们引用了非法指针，将可能导致崩溃。因此为了成功对数值进行整型到浮点型的转换，我已实现了浮点转换程序（如下），它在ConvolutionFilter 和DisplacementMapFilter中使用不同的混淆，并且不会造成在堆上出现的未知值被访问。

供选择的浮点类型转换程序

这解决了由于访问未知堆值而造成的崩溃问题，但是不幸的是，由于ConvolutionFilter matrix getter的实现，还存在很多与浮点值相关的问题，其会影响到利用程序的可靠性。在ActionScript 2中，所有数值类型的值会被存储为类型Number，它是在整型和双精度类型的指针之间的并集。native ConvolutionFilter matrix被存储为浮点型数组，但是会被复制进ActionScript数组中，以致于在matrix getter被调用时，用ActionScript可对它进行访问操作并且在处理过程中它的值会被计算为双精度值。接着，当浮点类型转换程序被调用时，会将双精度值计算得到浮点值。

将浮点值计算为双精度值通常会转换它的值，特定的案例除外。根据指定的浮点值，会有两种NaNs，一种是quiet NaNs (QNaNs) ，一种是 signalling NaNs (SNaNs)。QNaNs将不做任何事，但是在一些情景中，SNaNs会抛出浮点异常。在x86中，将双精度值计算为浮点值时总会导致QNaN（即使双精度值结果出自SNaN）来避免不可意料的异常问题。

因此如果指针的更低位恰好是SNaN，那么它将会被转换为QNaN，这意味着在它不应该是双精度值时，将会被设位（尾数的第一位，比特值22）。当vtable被读取时，则可避免该问题的出现，指针的第三个字节，它包含已经翻转的位，在内存中为未对齐状态，其可被读取来它真实的值。因此代码将进行未对齐读取（通过将Bitmap指针递增一进行vtable的再一次读取）并且如果浮点值恰好是SNaN时，会纠正整型值。

使用浮点值转换程序实现以上操作，vtable地址可被转换为整型值。现在我们需要使用该地址获取代码执行的权限。简单的方法就是移动指令指针来覆写某个对象的vtable（或某一个拥有vtable对象的指针）。这可以用DisplacementFilter BitmapData指针对ConvolutionFilter矩阵数组进行混淆操作来实现。

BitmapData对象组成一系列支援的（backing）native对象。ActionScript对象含有一个BitmapData native对象的指针，同时它也含有其它native对象的指针。一个类似的对象为bits对象，它含有确切的位图bits。该bits对象含有许多虚拟方法，当在BitmapData对象上进行任何操作时，经常会首先被调用。为了利用这一特性，利用程序会创建一个伪造的BitmapData对象，其带有一个伪造的bits对象的指针，并且会调用某个方法，接着调用一个方法，该方法将导致在伪造的bits对象上调用某个虚拟方法。


通过如上所描述过的setter，ConvolutionFilter.matrix性质可被使用来分配一个任意大小的buffer，该buffer用于浮点值的填充。通过一个DisplacementMapFilter并使用DisplacementMapFilter mapPoint属性混淆ConvolutionFilter来决定该buffer的位置，类似于完成对vtable位置的读取操作。

因为已分配的数据是不可变的，需要首先创建一个伪造的vtable对象和一个伪造的指向vtable的bits对象，然后创建一个指向已分配数据的伪造位图。

创建一个伪造的Bitmap

在有关ActionScript编程语言的应用中，伪造位图的某个引用可被检索到，凭借着用一个BevelFilter将某个对象混淆的方式，将DisplacementMapFilter对象的BitmapData对象设置为伪造位图的指针，并将颜色属性设置为指针值，与此操作相反的是完成对原BitmapData对象的位置的读取，vtable会被读出。该对象可被混淆回DisplacementMapFilter和通过调用mapImage getter可访问的BitmapData对象。无论何时，如果在某个含有一个虚拟调用（如setPixel32）的方法对象上被调用，那么在伪造的vtable中，该方法将会调入被指定的位置。

在这时，值得在更多地细节中了解伪造的vtable。在早前针对浮点型转换程序的讨论中忽略了与SNaNs相关的问题：写浮点值。在编写他们之前，ConvolutionFilter.matrix setter也让浮点值与双精度值之间互相转换，因此如果某个指针恰好是SNaN值，那么会被写入到matrix数组，即使它没有在原始值中被设定，占有的第23位也将会被设定，通过使用未对齐的写操作在某种程度上可使该问题得以避免。

一个SNaN指针在内存中的布局如下：

00: XX XX YY QQ XX ZZ 00 00

Where:

XX可以是从0到0xFF的任意值。

YY在占有的第五位上设为0并且没有其它的限制值。

在QQ上除了第7位可以是1或0外，其它所有位都设为1。

ZZ 是一个值，同时第七位会被设置为0，并且没有其它的约束。

它可被写为在内存中未对齐的32位浮点值如下:

00: 00 XX XX YY

04: QQ XX ZZ 00

08: 00 00 00 00

可以肯定的是如果原指针是一个SNaN值，那么在内存中没有未对齐的值将会是SNaN（因为如果原浮点值是SNaN，那么YY所占有的第5位将不设定）。即使该布局是如此，但是不可能用两个连续的指针进行该操作（除非他们都为SNaN）。

00: 00 XX XX YY

04: QQ XX ZZ 00

08: 00 XX XX XX

0C: QQ XX ZZ 00

10: 00 00 00 00

在0x08地址上的浮点值在第22到31位上的比特值是不受限制的，因此它可以是一个SNaN并且会是一个被不正确地写入的值。

因此不管它是SNaN或不是SNaN，将任意浮点类型数组的指针写到浮点型数组中都是可能的，但是它仅可被完成一次。在初始指针已经写入后，如果原始指针是一个SNaN值，那么所有额外的附加指针都应该是SNaN，反之亦然。该利用程序仅通过将一个指针写到任意ConvolutionFilter.matrix数组的方式来处理该限制。

对于伪造的Bitmap和伪造的Bitmap bits来说这一切都容易，因为他们仅需要含有一个指针。但是对于伪造的vtable来说却是一个挑战，即使它也含有一个指针。这会让其设定用于调用的参数以及对其进行调用时变得困难。

好的解决方案将会是移动到buffer中，在对vtable的调用之后该buffer完全可修改。幸运的是，BitmapData类（伪造的位图是仿造的对象）有某个方法，paletteMap，它创建类似的一个buffer。该方法有四个参数（redArray, greenArray, blueArray 和 alphaArray），在Actionscript中，它们都为数组。当该方法被调用时，它们会被转换为整型值并且会被复制到一个整型的native数组。接着，该数组（在恰当的偏移上用于输入的数组）的另一个带有四个指针的native方法被调用。该方法会进行虚拟调用，同时会转移进入伪造的vtable中。

作为一个初始的调用部分，数组指针被存储于x64寄存器r12,r13,r14和r15。这是非常有用的，因为它会让指针变为四个可控的buffer。这单一指针在伪造的vtable中并且为如下gadget：

mov rdi, r13

call [r12]

在r13上的buffer被设为字符串“gedit”，并且在r12上的buffer在Flash library中被设为一个gadget，它调用方法系统（不关注SNaNs）。当完成虚拟调用进入伪造的table后，将导致gedit被启动。

此处决定了利用程序是否有效，即使它不会完美地退出（当gedit退出后，Flash播放器会崩溃）。虽然将多个调用推入四个有效的buffer内，但是这对于修复来说微不足道。即使这被修复，也不能在Flash Player的破坏之后得以恢复（例如，如果tab被关闭，或者swf被刷新）。这是因为调用过滤对象的析构函数，由于用Bitmap对象对ConvolutionFilter matrix数组的指针进行混淆从而导致了崩溃的发生。这些对象在不同的堆上被分配，因此当其他的对象不被需要那么调用delete将会导致一次崩溃。通过将这些对象的混淆‘回’不错的状态不可能修正这个问题，因为在特定的bug中进行类型混淆会创建一个对象的副本，因此原不好的对象将保留，并且仍然需要被释放，因为在设定它之前，BitmapData对象和matrix对象setter会试图删除任意存在的对象。当利用程序在处理该过程时，可以避免这次崩溃的发生，并且和播放器那样正常打开，凭借着保留对那些对象的引用所以它们将不会被释放。当播放器被破坏时，仍然会发生崩溃。也就是说，在内存中通过纠正被混淆的类型，来避免在利用程序实现命令执行时造成的崩溃，或者在内存中通过将他们以正确的状态放置，或者将它们的析构函数指针设置为null。在该概念证明的利用程序中，这是不可实现的。

是什么让这个bug变得可靠？

通常会在存在类型混淆问题时，其bug可被利用。对于可靠的利用程序来说，存在一些经得起检验的因素。

使用被混淆的类型对象成员排布原始类型对象成员在对问题利用和可靠方面有着重大的影响。另一同样场景是存在漏洞的原对象成员扩展过去被混淆的类型成员并且他们的值是由越界内存决定的。此场景通过可利用，即使不可靠，因为确保堆块在内存中整齐排布是困难的。另一种可能性是在可被控制的对象上排布方法的限制，例如，原始类型对象的成员仅可被设置为受限制的值。该情景或者是可被利用的或者是基于特定bug的，也会导致存在类似于信息泄露的bug。

当类型转换被触发时，它总包括两种类型，当它被实例化时，存在漏洞的原始类型。并且在漏洞被触发后，它会变为被混淆的类型。

该bug被可靠利用具有可能性，但是需要特别‘幸运’的bug，这会导致恰当的成员拥有恰当的值。

该bug的另一方面在存在漏洞的函数发生混淆是当其不是时，在用某种方式使用类型被混淆的对象造成的类型混淆。笔记：在以上利用程序区域中，类型被混淆的MovieClip对象区域发生的操作时用0将其设为0.这防止了确切对过滤对象的调用，其中过滤对象可让利用程序变得不具有可靠性，因为过滤对象不需要被应用于一个没有像素的对象中。

与此同时，在该漏洞中，攻击者选择访问的对象成员以外的对象成员将不会被软件使用。另一个问题会影响类型混淆bug的可靠性与可利用性。有时没用的（从攻击者的角度看）如果攻击者不可能纠正他，那么成员会可造成崩溃。再次，用0将MovieClip设为0来防止该问题的发生，因为过滤方法不会被Flash访问，其中Flash用于没有像素的图片。

最后，重点在于能够保留原始和被混淆的对象，因为它防止垃圾收集，如果它在某个类型被混淆的对象上被调用，那么它总会崩溃。垃圾收集操作多半是通过用恰当的方法将对象成员进行“纠正”而进行的，如含有非法指针，如果没有正确的数据则会造成崩溃，并且通常对于某个攻击者来说不可能去纠正这个问题，因为垃圾收集的操作可以在任何时候发生时，因此任意非法的窗口都会是一个问题。唯一可靠的解决方案是当对象非法时，阻止进行垃圾收集操作。

总结

存在大量因素，包括原始布局和被混淆的对象成员，原始的布局情况以及当对象被使用时 可以影响所依赖的类型混淆bug。CVE-2015-3077算是特别高质量的漏洞，由于存在这些因素而让其可被利用。需要对该漏洞多达31次的触发操作来利用该漏洞：8次用于得到并设置需要用于利用的对象成员并且19-21次用于浮点类型转换，其取决于导致SNaNs发生的次数。这可能看来可能是大工程，该利用程序可成功实现，因为每一步骤都是促进利用程序成功实现目标的决定性因素并且不依赖于任何不被担保发生的行为。
