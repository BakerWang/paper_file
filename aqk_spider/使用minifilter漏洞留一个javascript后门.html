使用minifilter漏洞留一个javascript后门

2015-08-24 17:56:30 阅读：8580次 点赞(0) 收藏 来源： 360安全播报

除了令人难忘得生活经验以及个人的成长，我还从第23届DEFCON大会上了解到了Travis Goodspeed的POC||GTFO 0x08。

PoC||GTFO是最近几年深受0ld sch00l黑客喜爱的电子杂志，在大会期间，我从中读到的最优秀的文章就是一篇名为“利用编译器漏洞的可抵赖后门”的文章。在这篇文章中，作者向我们展示了一种可能的威胁，这个威胁模型是需要两个条件：一个可利用的编译器miscompilation漏洞；二是向目标的开源社区提交看起来不怎么像后门的patch。作者修改了Sudo 1.8.13的少量代码去利用Clang/LLVM 3.3的一个bug，最终导致提权。相比传统的编译器后门和传统软件后门（比如疑似NSA对Linux内核代码的植入），这种方式更加的隐蔽，因为它是针对特定版本的编译器下手。

这一切便让我联想到，我是否可以在javascript上使用同样的后门技术？在今天，我们几乎在任何地方都可以看到JS的存在（浏览器，服务器，Arduinos平台以及机器人，也许有一天还会出现在汽车上），但它是一个解释型语言，而不是一个编译型的语言。然而，通过缩小和优化JS来使文件变小和提高系统性能却是非常常见的。也许这种机制可以给我们足够的空间来使用JS minifilter来插入一个后门。

第一部分：寻找一个合适的minifilter漏洞

问题：在流行的JS minifilter中真的存在可以引起这些安全问题的漏洞吗？

回答：在进行了10分钟的搜索之后，我在UglifyJS中发现了一个漏洞，UglifyJS是一款热门的jQuery minifilter，它可以用来构建脚本程序，目前互联网上大约有70%的热门网站都使用了它。这个漏洞，在2.4.24版本中得到了修复，但是效果并不是非常的明显。让我们来看一看详细信息吧。

UglifyJS为了降低文件所占空间的大小，它会做很多事情。在默认情况下，其中一个压缩标志会对表达式进行压缩，例如：

!a && !b && !c && !d

这个表达式有20个字符。幸运的是，如果我们采用了De Morgan定律，那么我们就可以将其重写为：

!(a || b || c || d)

新表达式只有19个字符。非常好！但是，如果子表达式有非布尔值的返回值，那么De Morgan定律就不再适用了。比如说：

!false && 1

表达式将会返回数字1。从另一方面来说：

!(false || !1)

仅仅只会返回“真“。

所以如果我们能欺骗minifilter去错误地使用De Morgan定律，那么我们就可以让程序在文件缩小完成之后去做一些别的事情了。事实证明，完成对UglifyJS 2.4.23的欺骗并不难，因为如果重写的表达式比原始表达式的长度短，那么它就总会使用重写的表达式。（UglifyJS 2.4.24修复了这个问题，在子表达式进行重写之前，这个版本会确保子表达式返回的是布尔值。）

第二部分：在假设的验证码中构建后门

很好，我们已经找到了我们梦寐以求的minifilter漏洞了。现在，让我们来尝试利用这个漏洞！

我们假设你正在为某个公司工作，而且你想要故意地在公司的Node.js网站中增加某种漏洞。你现在的任务就是写一些能够验证用户身份令牌的服务器端javascript脚本，无论这些认证令牌是否过期。首先，你需要确保Node package使用了uglify-js@2.4.23，因为这个版本中的漏洞才是我们所真正关心的。

接下来，你需要写一个令牌验证函数，插入一堆看似有理的配置代码和用户验证功能，以此来迫使minifilter去压缩较长的（非）布尔表达式：

function isTokenValid(user) {
    var timeLeft =
        !!config && // config object exists
        !!user.token && // user object has a token
        !user.token.invalidated && // token is not explicitly invalidated
        !config.uninitialized && // config is initialized
        !config.ignoreTimestamps && // don't ignore timestamps
        getTimeLeft(user.token.expiry); // > 0 if expiration is in the future
    // The token must not be expired
    return timeLeft > 0;
}
function getTimeLeft(expiry) {
  return expiry - getSystemTime();
}

运行uglifyjs –c命令：

function isTokenValid(user){var timeLeft=!(!config||!user.token||user.token.invalidated||config.uninitialized||config.ignoreTimestamps||!getTimeLeft(user.token.expiry));return timeLeft>0}function getTimeLeft(expiry){return expiry-getSystemTime()}

在原始形式中，如果配置参数和用户检测通过了，timeLeft会在令牌过期时变成一个负整数。在minified形式下，timeLeft必须是一个布尔值。实际上，如果配置参数和用户检测通过了，timeLeft的值永远都会是true，除非getTimeLeft碰巧变成了0。

第三部分：jQuery后门

接下来，让我们利用这个minifilter漏洞来写一些jQuery后门。我们在实验中使用jQuery 1.11.3，这个版本是目前的稳定版本。

jQuery 1.11.3在进行文件缩小操作时将会使用grunt-contrib-uglify 0.3.2，它依赖于uglify-js ~2.4.0。所以uglify-js@2.4.23满足这个依赖条件，而且我们还可以在grunt-contrib-uglify中手动编辑package的内容来迫使它使用这个版本。

在jQuery中，有很多很多的地方可以使用De Morgan定律来重写和优化表达式。但这些都不会出发漏洞，所以我们必须自行添加一些。

后门补丁＃1:

首先，让我们在jQuery的.html（）方法中添加一个潜在的后门。补丁看起来非常奇怪和多余，但是我们可以让别人相信它并不会实际改变原本的功能。

在使用uglify-js@2.4.23完成文件缩小之后，jQuery的.html（）方法将会设置内部HTML值为“真”，而不会使用提供的值，所以测试失败。

然而，jQuery的维护人员很有可能会使用uglifyjs的修复版本。实际上，uglify-js@2.4.24的测试是通过了的，所以这个补丁看起来就没那么可疑了。

很好。现在，我们可以写一些傻瓜代码来触发这个后门程序了：

<html>
    <script src="../dist/jquery.min.js"></script>
    <button>click me to see if this site is safe</button>
    <script>
        $('button').click(function(e) {
            $('#result').html('<b>false!!</b>');
        });
    </script>
    <div id='result'></div>
</html>

下面的图片显示的是点击了那个按钮之后的结果：

正如我们所期待的那样，如果网站不安全，那么用户会被警告。这一切显得有些毫无意义，因为它并没有使用我们设定的后门。

后门补丁＃2:

第一个后门可能会很容易地被检测到，因为每一个使用它的人很可能都会注意到，HTML代码中很多表达式都被设置成了“true”。所以我们第二个后门补丁将只会在非正常情况下被触发。

可以这样说，在我们要设置后门的网站中，网页要包含下列HTML代码：

<html>
    <script src="../dist/jquery.min.js"></script>
    <button>click me to see if special event handlers are called!</button>
    <div>FAIL</div>
    <script>
        // Add a special event hook for onclick removal
        jQuery.event.special.click.remove = function(handleObj) {
            $('div').text('SUCCESS');
        };
        $('button').click(function myHandler(e) {
            // Trigger the special event hook
            $('button').off('click');
        });
    </script>
</html>

结论

我所演示的后门程序样例是经过特殊设计的。但是事实证明，这些后门的存在可能会让JS的开发者们感到担忧。尽管JS minifilter和C++编译器相比，JS minifilter就显得没有那么的复杂和重要了，但是JS minifilter的功能同样是非常的强大，而且目前有很多网站都会使用这个功能。
