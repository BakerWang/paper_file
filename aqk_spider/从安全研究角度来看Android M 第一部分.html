从安全研究角度来看Android M（第一部分）

2015-07-17 13:57:52 阅读：20990次 点赞(0) 收藏 来源： 360安全播报

小编注： Android M：Android的下一个版本

上一次Google I/O上，谷歌宣布并提供了Android M开发者预览。这篇文章总结了Android M这一平台上部分方面相关的安全性研究，并且尽可能的提示给你扩展信息。

自动备份

Android M对应用程序有自动备份功能。BackupManager服务会上传所有的私有应用程序数据到用户的Google Drive上（这不需要增加Google Drive配额）。

如果你是一个应用开发者并且你的应用包含有敏感数据，你不想在第三方云服务器上进行备份，你可以在应用的manifest中设置android:allowBackup属性为false。注意这也会禁用应用程序的ADB备份。如果你想要选择哪些内容可以被备份或者不能被备份，你可以使用备份方案配置文件设定具体规则来包含或者排除你应用程序中的私有数据备份。

你可以在命令行使用bmgr命令来调用BackupManager：adb shell bmgr run会初始化BackupManager，可以使用adb shell bmgr fullbackup <PACKAGE>来手动强制备份你的应用程序。同样可以使用adb shell bmgr restore <PACKAGE>来恢复应用程序数据。

当安装一个之前安装过但是后来已经卸载掉了的应用时，BackupManager会自动启动恢复过程来恢复之前应用程序自动云备份的私有数据。备份数据（和相关的元数据）可以使用adb shell bmgr wipe <TRANSPORT> <PACKAGE>命令进行清除，transports列表可以通过adb shell bmgr list transports进行获取。如果你想要打开transport组件的详细记录，可以使用adb shell setprop log.tag.GmsBackupTransport VERBOSE命令。

你可以在Mark Murpy的博客文章中阅读更多关于自动备份的内容。

权限

Android M推出了一个新的权限模型。它允许应用程序在运行时动态请求权限，而不是安装时授予所有权限。用户也可以在设置->应用->应用名->权限中撤销已经授予的权限许可。

并不是所有的权限都在运行时请求，PROTECTION_NORMAL保护级别的权限是在安装时授予，因为这些是低风险权限，例如RECEIVE_BOOT_COMPLETED或者VIBRATE。

如果应用程序SDK目标版本低于Android M，所有权限仍是在安装时被授予，但用户可以在任何时间手动撤销授权。

当SDK目标是Android M的应用所需的权限被撤销，应用会得到一个SecurityException，如果你忘了在应用程序的manifest申明权限也会得到相同的结果。如果应用程序SDK目标版本低于Android M，框架将只返回空数据或者直接忽略请求，以确保应用程序在没有对SecurityExceptions进行处理的情况下仍正常运行，不会崩溃。

新的权限模型也带来了新的实用ADB命令，我们可以使用adb install -g file.apk来授予应用程序所有在manifest中申明的权限。使用adb pm grant <PACKAGE> <PERMISSION>来授予特定权限，同样可以使用adb pm revoke <PACKAGE> <PERMISSION>来撤销特定权限。

你可以在Dave Smith的博客文章中阅读更多关于Android M权限模型的内容。

加密的连接

Android M包含有一些新工具来帮助开发者检测（和避免）在网络上发送未加密的数据。

开发者可以使用android.os.StrictMode API检测他们无意中的错误。StrictMode detectCleartextNetwork()关注未加密的网络套接字。StrictMode penaltyDropBox()会对违规触发情况进行记录。

StrictMode penaltyDeathOnCleartextNetwork()会在检测到异常时崩溃整个进程来避免意外的数据泄露。

还有一个新的应用程序级别的manifest属性：android:usesCleartextTraffic，默认为TRUE。

该属性用于应用的NetworkSecurityPolicy，并且声明是否允许未加密的网络流量。

当usesCleartextTraffic被设置为false，应用程序会在使用HTTP而不是HTTPS时崩溃。

你可以在John Kozyrakis的博客文章中阅读更多关于Android M加密连接的相关内容。

在Android的init中执行命令

通常OEM会修改init系统来执行他们自己的设置，现在Android M在Android init中提供了命令执行选项，可以给定参数指明支持用户和用户组。

Android init在过去是漏洞的重要来源。init的漏洞通常会导致权限提升。OEM会在他们自定义的init脚本中犯一些错误，因此命令执行功能应该被限制在selinux下。这是在root设备时需要注意的地方。

你可以在system/core/init/readme.txt中阅读官方文档。
