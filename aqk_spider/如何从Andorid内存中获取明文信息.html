如何从Andorid内存中获取明文信息

2015-08-27 15:25:16 阅读：7446次 点赞(0) 收藏 来源： 360安全播报

在这篇文章中,我将会给大家介绍一种我在安卓操作系统(我最喜欢的移动操作系统)上所使用的技术,但是这种技术的应用需要有正确的配置环境。我将给大家展示的是如何从安卓进程当中直接dump出内存数据。

我在文章的结尾谈到了“debug mode on apps”,而这一过程与我将要讨论的东西有些类似,但我将讨论的这个技术则显得更加的复杂,而且还需要与本地应用程序共同完成。

请注意:这一切听起来很棒吧?但是你需要root你的设备,而且还需要一个能够驻留在内存中的程序。

当然了,我找到了一个可供我使用的应用程序。

我在测试时所使用的这个应用程序并不会存储过多的真实数据,而且还有锁定功能,能够执行证书绑定,通过网络对比解锁码,所以该程序的数据并不会存储在本地设备上。除此之外,它并不会对设备是否root的状态执行任何形式的检测。

安装

好的,在实验开始之前,我们需要进行一些操作来配置好你的设备(已root)。你需要修改你的系统分区,然后将GNU调试器(gdb)添加进这个分区中。幸运的是,你不需要对其进行编译,因为安卓的原生开发套件(NDK)已经提供了一个预编译的版本给你了。

在得到原生开发套件(NDK)之后,将其中代码提取出来,然后找到合适你平台的gdbserver版本。由于我的实验设备是Nexus 4,所以我将会使用ARM版本。

操作步骤与将数据添加进root分区的步骤是一样的:

1.    上传文件;

2.    输入shell命令,然后root设备;

3.    重新加载/system目录,并设置为可读写(read/write);

4.    将文件复制到/system/xbin目录中(或者/system/bin);

5.    修改文件权限,以确保它可以正确执行;

6.    清理系统

或者你也可以这样:

push: gdbserver/gdbserver -> /sdcard/gdbserver
1 file pushed. 0 files skipped.

现在,我们需要一个gdb客户端来与服务器端进程进行通信。我们不能使用大多数Linux系统中的标准gdb进程,因为它们在我们的设备上并不能够正确地运行。所以我们需要对一个gdb进行自定义编译,这样才能确保它既可以运行在Intel架构的设备上,也能够支持ARM架构的设备。

首先,你需要从gnu.org上下载gdb源代码:(没错,我确实会在深夜写文章,我并不否认,因为这就是我!)

然后,我们就可以使用--target标记来给gdb指定目标了,并能够将其编译为一个可以识别ARM架构的进程:

[dave@jotunheim gdb-7.7]$ ./configure --target=arm-linux-gnueabi
[dave@jotunheim gdb-7.7]$ make

那么现在,我们的gdb目录中应该会有一个可供我们使用的gdb了,这个gdb可以运行在Intel架构的设备上,但是也能够识别ARM目标。

检测目标

现在,当gdb开始运行之后,我们就可以尝试从内存中读取数据了。实际上,我们真正感兴趣的是堆内存中的数据,因为设备上的原生程序和Dalvik虚拟机都会请求存储在这块内存中的数据。

我们可以使用设备的/proc伪文件系统来找到这些信息。比如说,我可以将设备密钥库中的内存数据dump出来。第一步,就是找出进程ID:

root@mako:/ # ps | grep key
keystore  228   1     4248   1680  c0624a6c b6f92838 S /system/bin/keystore

然后我们就可以进入/proc目录,并检查该进程的内存映射:

root@mako:/ # cd /proc/228
cd /proc/228
root@mako:/proc/228 # cat maps | tail
b6fc1000-b6fce000 r-xp 00000000 b3:15 159        /system/bin/linker
b6fce000-b6fcf000 r--p 0000c000 b3:15 159        /system/bin/linker
b6fcf000-b6fd0000 rw-p 0000d000 b3:15 159        /system/bin/linker
b6fd0000-b6fd1000 rw-p 00000000 00:00 0
b6fd1000-b6fda000 r-xp 00000000 b3:15 317        /system/bin/keystore
b6fdb000-b6fdc000 r--p 00009000 b3:15 317        /system/bin/keystore
b6fdc000-b6fdd000 rw-p 0000a000 b3:15 317        /system/bin/keystore
b7712000-b771f000 rw-p 00000000 00:00 0          [heap]
bee86000-beea7000 rw-p 00000000 00:00 0          [stack]
ffff0000-ffff1000 r-xp 00000000 00:00 0          [vectors]

通常情况下,我们能够从中发现该进程的代码,以及相关的库文件,还有一些有关该进程的重要信息:

l  堆-设备虚拟机或者系统内核用于存储数据而分配的一块内存空间

l  栈-函数调用过程中需要使用的内存空间

所以,在完成了上述操作之后,我们就可以看到堆内存的地址为0xb7712000 – 0xb771f000。

读取内存数据

好了,现在该开始读取内存数据了。操作步骤也许会有一点点复杂,但是当你熟悉了这些操作之后,你就会明白了,其实并没有你想象中那么难。具体操作步骤如下:

1.    开启gdbserver,并启用设备的进程监听端口;

2.    使用adb将设备端口信息转发到本地端口;

3.    使用第三方程序将本地端口数据转发至你需要运行gdb的设备上;

4.    通过gdb进行连接;

5.    dump出内存数据;

6.    Look for stuff;

7.    $$$ Profit $four$;

一、    开启gdbserver

这是整个操作过程中最简单的一个步骤了,当你安装好gdbserver之后,你可以将其添加至我们的进程中,并且为其指定一个用于监听的TCP端口。我使用的TCP端口为1234,具体信息如下:

1|root@mako:/proc/228 # gdbserver --attach :1234 228
Attached; pid = 228
Listening on port 1234

不要问我为什么进程ID和连接字符串是这种形式的,其实我也看不懂。

现在,我们要通过端口1234来监听网络数据了,但是由于iptables(iptables 是与最新的 3.5 版本 Linux 内核集成的 IP 信息包过滤系统)的存在,我们还需要进行下面的操作。

二、    使用adb转发端口

这将允许我们使用adb和我们的USB设备来为主机网络端口和设备之间进行桥接:


C:\Users\dave>adb forward tcp:1234 tcp:1234

然后,我们就可以通过TCP端口1234来与主机设备进行通信了。但还有一个问题:

C:\Users\dave>netstat -an | grep 1234
  TCP    127.0.0.1:1234         0.0.0.0:0              LISTENING

正如你看到的那样,进程只监听了localhost:1234,这也就意味着,如果我们的gdb运行在其他的主机或者虚拟机上,我们还需要进行下面的操作。

三、    使用第三方程序来转发端口

如果你准备在同一主机上运行gdb和adb,那么你就可以跳过这一步骤了。

我使用了一个名为“Port Forwarding for Windows”的非常老的程序来转发主机端口的信息:

为了操作方便,我们使用TCP端口1234。如果无法正常工作的话,请检查一下你的防火墙设置,因为防火墙经常会阻止用户的此类操作。

一切操作完成之后,我们现在就可以连接gdb了。

四、    连接gdb

那么,现在我要通过我的Windows操作系统来将Linux虚拟机和gdbserver(运行在我的Nexus 4上)连接起来了。在整个连接过程中,我需要使用那个我之前制作的自定义编译的gdb:

现在,我们已经完成了连接,然后我们就可以开始dump内存数据了。

五、    dump内存数据

我们可以使用“dump memory”命令来快速地完成这一部分的操作。

(gdb) dump memory /tmp/heapout 0xb7712000 0xb771f000
(gdb)

这一段代码将会把设备存储在堆内存中的所有数据全部写入/tmp/heapout 文件之中。
