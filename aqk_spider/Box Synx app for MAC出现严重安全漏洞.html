Box Synx app for MAC出现严重安全漏洞

2015-02-10 17:28:47 阅读：9547次 点赞(0) 收藏 来源： 360安全播报

Box Sync app的Mac用户需升级到4.0.6035版本。这款app泄露几个敏感信息如API 密钥、内部用户ID、URL、及密码。

Box Sync的问题

最 近，我重新审视了一下Mac版Box Sync app令人费解的混乱情况。如果你是一个Mac管理员，负责即使是很小的部署环境，你可能会知道，部署一个Box Syn app并且管理它的设置是多么地麻烦。它唯一的部署方法是一个应用程序包，它要是像一个正常的拖拽创建应用那样就好了：要将它部署到你的大规模部署中，你 只需将应用复制到/Applications里面，利用配置文件或MCX来配置由提供商公布的设置，那么一切就妥当了。但Box Sync的情况并非如此。Box提供了如何为大型客户部署Box Syn的指导，这些大型客户要求Mac管理员做到如下事情：

  • 将Box Sync 应用复制到/Applications

  • 将/Applications/Box Sync.app/Contents/Resources/com.box.sync.bootstrapper 复制到/Library/PrivilegedHelperTools

  • 运行执行首次运行安装程序的 sudo /Applications/Box Sync.app/Contents/Resources/com.box.sync.bootstrapper

对于许多Mac管理员来说，这些步骤会通过使用一个标准的Apple PKG得到更好的处理，他们这么做是正确的。执行手动复制操作并通过执行命令来手动完成安装的效果并不是很好。Apple安装程序包擅 长通过具体的所有权和权限（Box Sync安装进程的第一步及第二步）将文件置于具体的文件系统位置，由终端用户运行安装后命令进一步配置应用程序（第三步）。一旦Box Sync应用程序安装完毕，由于默认在后台会进行自动更新，情况会更加复杂。这对于家庭用户来说尚可，但在一个大型的部署环境中，管理员需要对工作流进行 维护。这个工作流会经过“测试”、“质量保证”、和“生成”阶段更新所有软件，这是有问题的。在改变这个应用程序的默认喜好方面，只能从Box获得少量信 息，并且这个应用程序的设置选项卡没有提供多少信息：

在我们的环境中，我们遵循了以上的测试/质量保证、生成流程，因此让Box Sync客户自我更新，并且不允许我们来确保它跟我们环境的兼容性是一个问题。由于Box缺少文档，我凭直觉检查了Box Sync包内是否藏有任何配置信息。

探险时刻

乍一看这个应用程序包的内容没有什么问题：

在框架（Frameworks）里面也没有什么有意思的东西，只有Python跟Growl。被捆绑的Python框架也能说得通（System Python能有什么问题呢？），因为我们知道Box服务就像Dropbox一样，部分是用Python写成的：

Resources文件夹里有很多PNG图像被当做文件类型、UI以及很多.nib文件的图标，所以我们会忽略它们。我们看到更多的Box标识是用Python写成的：

我们再检查一下include跟lib文件夹，这对于任何安装一个带有Python模块跟相关内容的Python应用程序来说，这些名字看起来都很熟悉：

似乎include文件夹只包含pyconfig.h头文件，所以我们来看看lib文件夹，它似乎包含的内容更多：

这 里最有意思的项目是site-packages.zip文件夹，这对于之前部署过Python应用的人来说也是熟悉的名称。OS X本身有一个名为site-packages的文件名称，位于/Library/Frameworks/Python.framework /Versions/2.7/lib/python2.7里。里面是第三方Python模块，可通过easy_install或者pip进行安装，并且可 被所有Python应用使用。让我们看看里面都有什么？未压缩的site-packages文件夹超过20 MB，包含51个子文件夹以及近140个.pyo文件。.pyo文件是Python优化字节码文件，实际上就像python运行一个.py程序文件时自动 创建的.pyc文件一样。深入观察文件夹内容时，我们可以看到更多情况。

深入观察狡兔三窟

多 数文件夹包含常用的Python模块，这些模块在Box Sync的数量级app中都会看到：许多PyObjc模块用来调用Apple框架、XML、SSL、JSON、SQLite、NTLM模块及多种其他看起 来不是太有意思的模块。而看起来真正有意思是box及boxsdk文件夹：

在 观察box文件夹时，我们的注意力被conf子文件夹所吸引，因为我们依然还在寻找如何将应用配置得更好一点的线索。现在，我们再来看看我们认为所 有.pyo文件都拥有的Python优化代码的文件类型。在谷歌上快速搜索一下“python pyo files”得知，他们通过使用从easy_install或pip上可获得的uncompyle2工具恢复到常规的Python代码。一旦被安装，我们 会对整个网站——包文件夹的内容运行uncompyle2，而这个文件夹通过-r来对它进行递归处理。如果还有其他地方潜伏着有趣的配置选项，我们一定能 够找到。

$ easy_install uncompyle2
$ uncompyle2 -o /tmp/site-packages-decomp -r -p 8 /Applications/Box Sync.app/Contents/Resources/lib/site-packages

decompiled 1 files: 1 okay, 0 failed, 0 verify failed

<snip>


处 理过程结束后，我们来看看/tmp/site-packages-decomp的内容，并且看看uncompyle2生成的带有.pyo_dis扩展的文 件，且扩展目前仅仅是.py文件。我们能够对它们进行全部重命名，但现在任何文本编辑器都会读取它们，所以我们先不管。

深入探索Plist

在Textmate或BBEdit等app中打开/tmp/site_packages-decomp文件是搜索整个与喜好如一个plist文件有关的代码库最便捷的方式。首先我们会使用Textmate2来搜索所有包含“plist”文本的所有文件：

这件事儿不太难，对吧？似乎site-packages/box/conf/base.pyo_dis上的文件提到了一个名为/Library/Preferences/com.box.sync.plist的文件：


conf.set(u'preferences.mac_plist_file.path', u'/Library/Preferences/com.box.sync.plist')


找 到了！正常情况下，这个文件不会在为用户安装且配置了Box Sync 的系统中找到，所以这是一个非常好的开始。进行更多搜索之后，发现了configuration.py中包含着 _overrideable_settings列表，而configuration.py中有一个名为auto_update.enabled的key！ 这正是我们在寻找的设置。它在设置列表中名为“可重写设置”的包含进一步增强了我们的信息。为了测试这个设置真的起作用，我们可利用defaults编写 一个新的/Library/Preferences/com.box.sync.plist文件，如：


$ sudo defaults write /Library/Preferences/com.box.sync.plist auto_update.enabled -bool False


确 实，当通过安装一个带有/Library/Preferences/com.box.sync.plist喜好文件（将 auto_update.enabled key设置为False）的旧版Box Sync时，这款app确实不再尝试自动更新。任务完成了，打道回府吧？呃……几乎上可以启程了。

这时，开始动真格了。

一旦我开始在base.py及auto_update_release.pyo文件中滚动了几下并且看到里面还有什么时，我的反应可用下图来表示：

结果显示，Box开发团队将很多敏感信息内嵌到属于conf模块中的文件中。进行快速扫描之后，发现了如下看起来很敏感的key/value对，如：


app.remote_control_port
app.remote_control_auth_key
api_key #(Mac/Windows specific)
client_secret #(Mac/Windows specific)
upload_only_auth_token_for_log_folder
auto_update.internalqa_user_ids
mac_api_key
win_api_key
mac_client_secret
win_client_secret
au_password
v1_internal_admin_login_url
s3_access_key
s3_secret_key
jenkins_url


这可能并不是一件好事。因为机器人的存在，scan Github及 其他公共版本会控制以及无意间登录API密钥以及秘密的服务，Box可能并不想在Box Sync应用中披露所有的这些信息。要澄清一点，我并不想利用我所找到的任何信息来获取访问任何Box系统的权限。同时，我不会将所有的源代码或完整的 key/value对进行公布，这都留给有好奇心的读者去探索吧。在1月初期，在意识到我所发现的内容之后，我将这个问题报告给Box安全团队，并且在得 到对问题严重性的确认之后，我被要求删除披露，为Box开发团队提供修复时间。2月6日，我发现更新版本4.0.6035已发布，并且应该已将问题修复。 现在既然新版本已经发布更新，那么我将研究结果予以发布，希望给Mac管理员或其他使用或部署Box Sync的人员提一个醒，以确保4.0.6035更新被尽快更新。目前尚不清楚是否有其他人在我之前发现了这个问题，也不清楚这些信息是否已被用来访问 Box的客户信息。这对于使用管理软件更新工作流的环境尤为重要，只有当管理员采取特定的措施后，才能阻碍这个自动更新。

结论

希望这条信息能对Mac管理员及Mac个人用户起到作用。我再次重申，每个Boc Sync用户都应当确保他们所安装是4.0.6035版本或以上。
