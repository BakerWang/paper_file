Windows DLL注入基础分析与防护

Sam • 2017年1月17日 • 网络安全 • 阅读 1992

0x00 定义

DLL注入就是将一个DLL放进某个进程的地址空间里，让它成为该进程的一部分。通常插入的代码是以动态链接库（DLL）的形式，因为DLL是在运行时根据需要加载的。然而，这并不意味着我们不能以其他形式（可执行文件，手写等）注入程序集。要注意的是你需要有一个适当的系统特权级别。

0x01 概述

Windows API实际上提供了很多功能，允许我们附加和操作到其他程序以进行调试。 我们将利用这些方法来进行我们的DLL注入，DLL注入分为四个步骤：

  1. 附加到进程
  2. 在进程中分配内存
  3. 复制DLL或DLL的路径到进程的内存和确定适当的内存地址
  4. 引导进程执行你的DLL

上面的每一个步骤可以通过使用下图总结的一种或多种编程技术来实现，重要的是要了解每个技术的细节/选项。

 

执行起点

当指示目标进程启动我们的DLL时，我们有几个选项（例如CreateRemoteThread（）、NtCreateThreadEx（）等）。不幸的是，我们不能仅仅只为这些函数提供我们的DLL的名称，而是要提供一个内存地址来开始执行。我们执行分配和复制步骤以获得目标进程的内存空间，并将其准备为执行起点。

有两个普遍的执行起点：LoadLibraryA（）和跳转到DllMain。

LoadLibraryA()

LoadLibraryA() 是一个kernel32.dll函数，用于在运行时加载DLL、可执行文件和其他支持库。它需要一个文件名作为其唯一的参数，使一切可以正常工作。这意味着我们只需要为我们的DLL分配一些内存，并将我们的执行起点设置为LoadLibraryA（）的地址，从而提供路径作为参数的内存地址。

LoadLibraryA（）的主要缺点是它用程序注册加载的DLL，因此可以很容易地检测。另一个有点烦人的警告是，如果一个DLL已经被LoadLibraryA（）加载一次，它将不会执行该DLL。

跳转到DllMain（或另一个入口点）

LoadLibraryA（）的另一种方法是将整个DLL加载到内存中，，然后确定到DLL入口点的偏移量。 使用这个方法可以避免注册DLL与程序（隐身），并重复注入进程。

0x02 附加到进程

首先，我们需要一个进程句柄，以便我们可以与它进行交互。 这是通过OpenProcess（）函数完成的。我们还需要请求某些访问权限，以便我们执行以下任务。我们要求的特定访问权限（因Windows版本而异），但以下内容最适用：

hHandle = OpenProcess( PROCESS_CREATE_THREAD | 
                       PROCESS_QUERY_INFORMATION | 
                       PROCESS_VM_OPERATION | 
                       PROCESS_VM_WRITE | 
                       PROCESS_VM_READ, 
                       FALSE, 
                       procID );

0x03 分配内存

在我们将任何东西注入另一个进程之前，我们需要一个地方来放置它。 我们将使用VirtualAllocEx（）函数来达到这个目的。

VirtualAllocEx（）需要分配的内存量作为其参数之一。如果我们使用LoadLibraryA（），我们将为DLL的完整路径分配空间，如果我们跳转到DllMain，我们将为DLL的全部内容分配空间。

DLL路径

为DLL路径分配空间稍微减少了你需要写入的代码量，它还要求您使用具有一些缺点（如上所述）的LoadLibraryA（）方法。但LoadLibraryA（）却是一个非常受欢迎的方法。

使用VirtualAllocEx（）并分配足够的内存以支持包含DLL路径的字符串：

GetFullPathName(TEXT("somedll.dll"), 
                BUFSIZE, 
                dllPath, //Output to save the full DLL path
                NULL);

dllPathAddr = VirtualAllocEx(hHandle, 
                             0, 
                             strlen(dllPath), 
                             MEM_RESERVE|MEM_COMMIT, 
                             PAGE_EXECUTE_READWRITE);

完整的DLL

为完整的DLL分配空间需要更多的代码，但它却更可靠，不需要使用LoadLibraryA（）。

首先，使用CreateFileA（）打开DLL的句柄，然后使用GetFileSize（）计算它的大小，并将其传递给VirtualAllocEx（）：

GetFullPathName(TEXT("somedll.dll"), 
                BUFSIZE, 
                dllPath, //Output to save the full DLL path
                NULL);

hFile = CreateFileA( dllPath, 
                     GENERIC_READ, 
                     0, 
                     NULL, 
                     OPEN_EXISTING, 
                     FILE_ATTRIBUTE_NORMAL, 
                     NULL );

dllFileLength = GetFileSize( hFile, 
                             NULL );

remoteDllAddr = VirtualAllocEx( hProcess, 
                                NULL, 
                                dllFileLength, 
                                MEM_RESERVE|MEM_COMMIT, 
                                PAGE_EXECUTE_READWRITE );

0x04 复制DLL /确定地址

我们现在可以将DLL（路径或内容）复制到目标进程空间。

现在我们在目标进程中分配了空间，我们可以将我们的DLL路径或完整的DLL（取决于您选择的方法）复制到该进程中。 我们将使用WriteProcessMemory（）进行：

DLL路径

WriteProcessMemory(hHandle, 
                   dllPathAddr, 
                   dllPath, 
                   strlen(dllPath), 
                   NULL);

完整DLL

我们首先需要将我们的DLL读入内存，然后再将其复制到远程进程。

lpBuffer = HeapAlloc( GetProcessHeap(), 
                      0, 
                      dllFileLength); 

ReadFile( hFile, 
          lpBuffer, 
          dllFileLength, 
          &dwBytesRead, 
          NULL );

WriteProcessMemory( hProcess, 
                    lpRemoteLibraryBuffer, 
                    lpBuffer,  
                    dllFileLength, 
                    NULL );

确定执行起点

大多数执行函数需要一个内存地址开始，所以我们需要确定这个地址是什么。

DLL路径和LoadLibraryA（）

我们将搜索我们自己的进程内存中的LoadLibraryA（）的起始地址，然后将它作为参数传递给我们的执行函数，DLL的内存地址为Path。 要获取LoadLibraryA（）的地址，我们将使用GetModuleHandle（）和GetProcAddress（）两个方法：

loadLibAddr = GetProcAddress(GetModuleHandle(TEXT("kernel32.dll")), "LoadLibraryA");

完整的DLL和跳转到DllMain

通过将整个DLL复制到内存中，我们可以避免注册我们的DLL与进程，进行更可靠注入。 这样做当它被加载在内存中很难获得我们需要的DLL的入口点。 幸运的是，Stephen Fewer让我们轻松做到这一点。他率先推出了反射式DLL注入技术，与现有方法相比，提供了更大的隐身性。包含在他的ReflectiveDLLInjection Inject项目中的LoadRemoteLibraryR（）函数完全实现了这一点，但是它将我们的执行方法限制为CreateRemoteThread（）。 因此，我们将使用GetReflectiveLoaderOffset（）确定我们的进程内存中的偏移量，然后使用该偏移量加上我们写入DLL的目标进程中的内存的基地址作为执行起点。 这里需要注意的是，我们注入的DLL必须符合适当的包含和选项，以使其与ReflectiveDLLInjection方法一致。

dwReflectiveLoaderOffset = GetReflectiveLoaderOffset(lpWriteBuff);

0x05 执行DLL！

到这里，在内存中已经有我们的DLL，知道我们想要开始执行的内存地址。下一步准备执行它， 有几种方法可以做到这一点。

CreateRemoteThread()

CreateRemoteThread（）函数可能是最广泛使用的方法。在大多数情况下它是非常可靠的，但你可能想使用另一种方法以避免检测，或者如果微软更改某些事情导致CreateRemoteThread（）停止工作。

由于CreateRemoteThread（）是一个非常成熟的函数，你在使用它时会有比较大的灵活性。 例如，使用Python做DLL注入！

rThread = CreateRemoteThread(hTargetProcHandle, NULL, 0, lpStartExecAddr, lpExecParam, 0, NULL);
WaitForSingleObject(rThread, INFINITE);

NtCreateThreadEx()

NtCreateThreadEx（）是一个未记录的ntdll.dll函数。  没有文档的函数的麻烦是他们可能会随着微软的决定而消失或改变。但当Windows Vista的会话分离影响了CreateRemoteThread（）DLL注入时，NtCreateThreadEx（）却很好用的。

有关此方法的详细信息如下所述：http://securityxploded.com/ntcreatethreadex.php

NtCreateThreadEx（）是一个更复杂的调用，需要一个特定的结构传递给它，另一个从它接收数据。下面是详细的实现：

struct NtCreateThreadExBuffer {
 ULONG Size;
 ULONG Unknown1;
 ULONG Unknown2;
 PULONG Unknown3;
 ULONG Unknown4;
 ULONG Unknown5;
 ULONG Unknown6;
 PULONG Unknown7;
 ULONG Unknown8;
 }; 


typedef NTSTATUS (WINAPI *LPFUN_NtCreateThreadEx) (
 OUT PHANDLE hThread,
 IN ACCESS_MASK DesiredAccess,
 IN LPVOID ObjectAttributes,
 IN HANDLE ProcessHandle,
 IN LPTHREAD_START_ROUTINE lpStartAddress,
 IN LPVOID lpParameter,
 IN BOOL CreateSuspended,
 IN ULONG StackZeroBits,
 IN ULONG SizeOfStackCommit,
 IN ULONG SizeOfStackReserve,
 OUT LPVOID lpBytesBuffer
);

HANDLE bCreateRemoteThread(HANDLE hHandle, LPVOID loadLibAddr, LPVOID dllPathAddr) {

 HANDLE hRemoteThread = NULL;

 LPVOID ntCreateThreadExAddr = NULL;
 NtCreateThreadExBuffer ntbuffer;
 DWORD temp1 = 0; 
 DWORD temp2 = 0; 

 ntCreateThreadExAddr = GetProcAddress(GetModuleHandle(TEXT("ntdll.dll")), "NtCreateThreadEx");

 if( ntCreateThreadExAddr ) {
 
  ntbuffer.Size = sizeof(struct NtCreateThreadExBuffer);
  ntbuffer.Unknown1 = 0x10003;
  ntbuffer.Unknown2 = 0x8;
  ntbuffer.Unknown3 = &temp2;
  ntbuffer.Unknown4 = 0;
  ntbuffer.Unknown5 = 0x10004;
  ntbuffer.Unknown6 = 4;
  ntbuffer.Unknown7 = &temp1;
  ntbuffer.Unknown8 = 0;

  LPFUN_NtCreateThreadEx funNtCreateThreadEx = (LPFUN_NtCreateThreadEx)ntCreateThreadExAddr;
  NTSTATUS status = funNtCreateThreadEx(
          &hRemoteThread,
          0x1FFFFF,
          NULL,
          hHandle,
          (LPTHREAD_START_ROUTINE)loadLibAddr,
          dllPathAddr,
          FALSE,
          NULL,
          NULL,
          NULL,
          &ntbuffer
          );
  
  if (hRemoteThread == NULL) {
   printf("\t[!] NtCreateThreadEx Failed! [%d][%08x]\n", GetLastError(), status);
   return NULL;
  } else {
   return hRemoteThread;
  }
 } else {
  printf("\n[!] Could not find NtCreateThreadEx!\n");
 }
 return NULL;

}

现在我们可以调用CreateRemoteThread（）：

rThread = bCreateRemoteThread(hTargetProcHandle, lpStartExecAddr, lpExecParam);
WaitForSingleObject(rThread, INFINITE);

0x06 边注：DLL代理/ DLL劫持

注意，DLL注入不同于DLL代理和劫持。出于某种原因，人们倾向于混淆这些。后者模拟合法的DLL，本质上“欺骗”应用程序加载它，而前者在其运行时将一个DLL插入进程。

DLL代理最常见的假设是你可以完全控制应用程序的安装目录。 “攻击者”重命名合法的DLL并将自己的DLL复制到安装目录中。当应用程序运行时，它加载攻击者的DLL（因为它的名称正确），然后攻击者的DLL将在函数调用中继到合法的执行。DLL代理最常用于系统的实际所有者作为扩展应用程序功能的方法。 例如，DLL代理在游戏中是流行的。 很多人使用这种技术来修改游戏功能作弊或其他种类的乐趣。”Spy”应用程序还利用DLL代理来尝试捕获用户提供的应用程序值。

DLL劫持类似于代理，但不同之处在于劫持通常滥用Windows的DLL搜索顺序，以危及系统（或以其他方式控制应用程序的流程）。 它通常不需要攻击者对应用程序的安装目录具有写入权限，而是应用程序启动的目录。在应用程序试图调用不存在的DLL或者攻击者能够将恶意DLL放置在与启动脆弱的应用程序的文件相同的目录中的情况下，将加载攻击者的DLL并且将实现代码执行 。这是因为Windows从应用程序加载的当前目录中搜索应用程序DLL。
