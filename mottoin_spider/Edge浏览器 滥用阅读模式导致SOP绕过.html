Edge浏览器：滥用阅读模式导致SOP绕过

Exp1oi7ss • 2017年4月18日 • 网络安全 • 阅读 370

前言

Microsoft Edge团队最近推出了阅读模式，这个功能可以消除网页中的多余内容，让你安心阅读，无需分心。这对我们来说不是新鲜事，但是我从来没有测试这种伪协议，我想了解这种伪协议如何在Edge上工作。你可以先观看视频PoC利用成功的效果后，继续阅读下面。

概述

想使用阅读模式的功能，首先要加载网站，然后点击阅读视图按钮（图书图标）。

这功能可以让我们更容易的阅读网页。

但是，这个网页阅读的真实URL位置是什么？打开开发工具（F12）并在控制台中键入location.href。很明显，Edge在URL上添加了read: pseudo-protocol协议。

以下漏洞适用于所有Edge版本，但PoC本身是针对Edge 15（创建者更新后）构建的。要在旧版本上执行，必须进行更改，且确保阅读模式能够使用。

阅读模式是一种内置资源

阅读模式下的网页与真实网站无关。我们看一下源码（按CTRL U），我们会看到没有原始页面的跟踪，实际上这是一个托管在文件系统中的内置资源：

C:\Windows\SystemApps\Microsoft.MicrosoftEdge_8wekyb3d8bbwe\Assets\ReadingView

Edge解析原始网页的内容，删除iframe/scripts和其他html标签，终于使得它在内部阅读查看HTML托管的iframe。但所有这些改变都只是在后台发生，用户最终会产生还在原始网站上的错觉，因为地址栏并没有改变。

如果Edge通过在真实URL前面设置“read：”协议，在阅读模式下呈现网页，我们可以从脚本中执行吗？我们可以加载在自动读取模式的任何网址？

强迫任何网站进入阅读模式

我们来看看是否可以强制让任意的URL通过前面的read: protocol在阅读模式下呈现 。

location.href = "read:http://www.cracking.com.ar"; // prepending read: does the trick

这样做的效果非常好，而且引起了我的注意：地址栏中的URL是crack.com.ar，但是rendered的内容来自brokenbrowser.com。就是说，如果我们打开crack.com.ar这个网站，我们看到的其实是location.replace到brokenbrowser.com，但Edge没有更新地址栏！

漏洞＃1 – 在阅读模式下，当script/http重定向发生时，Edge不会更新地址栏。

一个有趣的重定向

这意味着我们可以欺骗任何具有允许重定向或者是已经被我们控制的任何被重定向到我们网站的网站。例如，如果我们可以使google.com重定向到一个恶意的页面，那么用户将会认为内容来自谷歌，实际上它将来自evil.com。

顺便说一下，欺骗google不难，因为所有的organic结果都是重定向到目标网站的链接。例如，谷歌已经从crack.com.ar索引了一个“crack-01.html”页面，如果我们找到重定向到该页面的organic链接，我们可以控制它，因为在我的服务器中，我可以随意更改它！让我们打开Chrome搜索，用google搜到的一个链接并把它重定向到我的服务器（crack.com.ar）。请记住：我们的目标是找到可以重定向到crack.com.ar的Google网址，crack.com.ar网站是我们控制的网站。

在阅读模式下重定向

现在我们有一个已经把google.com.ar重定向到crack.com.ar的URL。在crack.com.ar中的网页有一段文字写着“Not really Google” ，所以我们可以很容易地确定内容真正来自哪里。以下是使用前面的阅读协议read: protocol的Google重定向，在Edge中打开，如下所示：

read:https://www.google.com.ar/url?sa=t&rct=j&q=&esrc=s&source=web&cd=1&cad=rja&uact=8&ved=0ahUKEwiRx

我们成功的实现了欺骗。但是问题来了，我们正在阅读模式中，这意味着我们没有办法控制我们页面的外观。记住：在渲染我们的tricky页面之前，Edge剥离了大量的HTML内容。例如，iframe和script被删除，并且JavaScript链接也不起作用了（多亏了在我们的内容之前呈现的META CSP标签）。我们如何自定义此页面并摆脱那个淡黄色的背景？我们如何在这里运行脚本？

在阅读模式下运行脚本

当我们在阅读模式时，Edge会尽力保持我们阅读的是静态内容，这意味着脚本不允许运行，iframe被丢弃等。换句话说，我们的内容最终看起来更像是一本书，而不是一个网页。但是，我们会尝试打破这种一切都看起来都是冻结的静态阅读模式的障碍。

我手动测试了几个html标签，如iframe/script/meta， 但是它们都被丢弃删除了。然后，我尝试了一个object/html标签，令人惊讶的是，它工作！这比预期更容易，object/html标签几乎可以模拟iframes全部功能：它们都是能够运行脚本的html容器。

漏洞＃2 -在阅读模式下渲染时Microsoft Edge边缘不会删除对象标签。

所以，如果我们在crack.com.ar中添加一个对象标签，使我们的tricky页面触发一个提示，那么它看起来很有说服力。

<!-- prompt.html does a window.prompt with the hard coded "google.com needs..." message -->
<object data="http://www.cracking.com.ar/prompt.html"></object>

现在，Edge会认为首页的起源是google.com.ar（不是真的，这是crack.com.ar），但是object/html的起源是crack.com.ar（这是真的）。我们可以抛出提示/警报，但是我们无法访问顶部。

假设我们要将首页的背景颜色更改为白色，或者写一些东西来使攻击更具说服力，我们将需要绕过相同的原始策略或设置顶级URL，而不用更改地址栏。我们试试前者，绕过SOP。

在<object>框之外思考

我们如何代表顶级域来呈现任意的html代码，并且我们可以访问到它？这里可以使用data uri。我们将在data uri中呈现html，而不是把在crack.com.ar中托管的内容渲染，就像这样：

<!-- object rendering a data uri -->
<object data="data:,<script>alert(top.location)</script>"></object>
<!-- ACCESS DENIED data uris have their own unique origin -->

Edge不允许我们从这个data uri访问任何其他文档，这是正确的！所有浏览器都将data uri作为与creators不同的独特起源，但在Edge上，这个限制很容易绕过：在加载页面之后的self-document.write足以将我们的位置设置为我们的parent。

漏洞＃3 -在data uris中，self document.write将起始设置为与parent/creator相同。

<object data="data:,<script>
window.onload = function()
{ // Executing a document.write in a data uri after the onload
  // changes the location of the object to its parent URL.
  document.write('<script>alert(top.location.href)<\/script>');
  document.close();
}
</script>"></object>
<!-- Now we have the same location as our top -->

我们现在正在访问Google的顶级域名。在这一点上，我们可以完全访问渲染阅读模式的内部html代码，而不用改变任何东西，我们大胆的来做一个top.document.write，来摆脱淡黄色的背景。

<object data="data:,<script>
window.onload = function()
{
  document.write(
    '<script>'+
        'top.document.write(\'Trust me, we are on Google =)\');'+
        'top.document.close()'+
    '<\/script>');
  document.close();
}
</script>"></object>
