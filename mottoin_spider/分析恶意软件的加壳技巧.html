分析恶意软件的加壳技巧

M0tto1n • 2017年7月11日 • 安全报告 • 阅读 272

前情提要

恶意软件作者通常会使用一些技巧来规避检测和分析，最流行的方法之一就是使用加壳工具对恶意软件进行加壳包含。加壳工具是一种压缩、加密、和／或修改恶意文件格式的工具。加壳也可以被用于合法目的，比如为了防止程序被破解或复制。所有这些技巧都可以起到降低被反恶意软件产品检测到的机会，有助于逃避安全人员的分析或提升分析难度。

恶意软件被加壳之后，安全人员更加难以识别恶意软件的行为，所需的分析时间也相应的有所增加。脱壳是针对此类恶意软件的第一个挑战，然而加壳工具多种多样，复杂度也不尽相同。

加壳的类型

加壳可以起到简单的保护二进制代码的作用，攻击者更倾向于使用加壳器而非直接在代码内实现保护。一些先进的恶意软件由有组织的网络犯罪团体进行编码并维护，可能会使用到定制的加壳工具，或者在恶意文件内部实施复杂的防护措施。

对加壳器而言，stub（包含解压或解密程序的一段代码）扮演了加载的角色，在恶意软件之前被执行。

图1:加壳器加密／压缩数据过程图

首先，原始文件被传递到加壳程序中，以一个新的.exe被存储在一个Packed Section （加壳部分）中，一旦文件被运行，存储在加壳文件中的压缩 stub 程序将解压缩对应的Packed Section，然后将原始.exe文件加载到内存中。

在这种情况下，原始入口点（即程序启动的内存地址）被迁移到 packed Section中，分析人员检索到它才能恢复原始文件。

其他的加壳程序类似于一个代理器，对导入地址表（import address table ，IAT）进行保护。IAT钩子主要使用的IAT表进行API HOOK， 程序运行过程中，IAT被动态解析并在需要的时候被利用，恶意软件使用这种保护增加其被脱壳的难度。事实上在IAT不完整的情况下内存转储也无法正常工作。没有完整的 IAT，很难正确的分析恶意软件。

图2:加壳器混淆API的过程

加壳 Stub 扮演着代理的角色，拦截每次API 调用。调用请求从 API 转换到Stub中，并由有效载荷运行。

在这种情况下，恶意软件分析员必须通过定位函数并跟踪返回程序来修复导入的情况。Stub将重新执行原始的 API，在某些情况下，API的地址位于 EIP 注册表。

虚拟化代码也是一种有效的加壳技术，使用了虚拟化代码、模拟处理器的行为去运行原文件。这种技术相当于在执行代码和它的行为之间增加了一个抽象层。原始代码被转换成虚拟机的字节码。对恶意软件分析员而言，这种技术是最难分析的技术之一。

图3:代码虚拟化过程示意图

原始文件在转换时显示为 X86 汇编语言。文件被虚拟化保护后，转换后的代码存储在虚拟机的字节码中。

为了消除这种保护，恶意软件分析员必须研究虚拟软件的行为、检索字节码，用于分析原始文件。

其他还有一些结合了反调试或反沙箱技术，以及增加更多功能以保护恶意软件的加壳技术。某些恶意软件会使用多层防护措施以抵抗检测和分析。为了分析这类恶意软件，分析员必须除掉每一层保护壳，直到恢复原始文件为止。最快的恢复技术是运行恶意软件并直接从内存中转储。不过实践起来会有很多困难，因为很多加壳器附带了反转储技术。

使用加壳器依然是抵御反恶意软件产品的有效途径，攻击者偏好使用加壳器对自己的恶意软件进行保护。单纯的依靠手工分析很难成功，需要结合有效的工具，在不同模式下不断的调试，才能更加有效的对恶意软件进行脱壳并进一步的分析。
