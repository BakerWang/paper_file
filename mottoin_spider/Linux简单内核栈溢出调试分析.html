为什么出错呢？

这个就是因为SMEP了。

SMEP(Supervisor Mode Execution Protection)

查看维基的说明如下

意思是，如果设置了这个寄存器位，程序不能运行比自己的ring还高的地址。简单说就是内核的程序不能随便运行用户态的代码。

（it is documented in the newest release (May 2011) of the Intel manual 3A sections 2.5 (CR4.SMEP flag) and 4.6 (per memory page settings).）

相关的还要SMAP（虽然这里不涉及）：

SMAP（supervisor mode access prevention，防止超级用户访问）机制。SMAP（supervisor mode access prevention），它将阻止处理器运行在 CPL < 3 权限时访问属于 user 页面里的数据（即：各级页表项最终结合的 U/S 权限位为 1）。 也就是：处理器运行在 CPL < 3 权限下只能访问属于 supervisor 的页面

所以这次不能成功提权了，尝试内核rop

现在呢，我的想法就是关掉SMEP，参照前面的SMEP的介绍，我们只需要将CR4寄存器的第20bit位置位一下就可以了，所以汇编代码如下

mov eax,cr4
btr eax,20
mov cr4,eax
add rsp,0x8//平衡堆栈
jmp eviltest

btr指令的操作过程：

CF = Bit(BitBase, BitOffset);

Bit(BitBase, BitOffset) = 0;

总结

我的rop还不够完美，因为需要依赖test_data的地址,暂时也想不出比较好的方法获取，一开始卡很久也是在前面查看test_data结构体的时候，尝试查看内核消息导致不能查看，然后耗费了很久.希望如果你有什么疑问,mail我^0^ 。同时，慎用memcpy。

其他问题

从用户跳回内核的时候,我并没有使用iret,而是使用的ret,但是显然还是成功跳回了内核,我不清楚为什么.SMEP也只是让内核不能跳转到用户层,但是用户层跳转到内核总不能随随便便跳吧?

如果调试机在用户层的提权代码停留时间过长,会导致莫名其妙的跳转错误,猜猜可能是cpu时间判断上觉得该模块使用超时,然后进行了线程切换,导致内核栈被破坏???测试了好几次都是这样,如果强行修改RIP也是如此,试着只在ret处下断,然后continue,还是会出错,单步更容易出错,暂时想不通为什么.
